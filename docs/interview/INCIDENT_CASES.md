# PaperAgent 故障案例讲解

## 案例一：客户端超时后重复提交

**现象**：客户端没有收到首次 202，使用同一 Idempotency-Key 重试。

**处理**：后端在事务中查到相同 Key 和相同 Request Hash，返回原 Task ID，`reused=true`，不会创建第二个任务。

**面试重点**：幂等不是前端约定，而是数据库唯一约束、规范化 Hash 和事务共同保证。

## 案例二：相同 Key 被错误复用于不同请求

**现象**：客户端复用旧 Key，但修改了问题或约束。

**处理**：Request Hash 不一致，返回 409。

**面试重点**：不能因为“Key 已存在”就直接返回旧结果，否则会把错误请求静默绑定到旧任务。

## 案例三：模型返回不合法结构

**现象**：Provider 返回 JSON，但缺少生产 Schema 必填字段。

**处理**：不进入 Graph State；最多执行一次 Repair。Repair 仍不合法则记录类型化失败。

**面试重点**：格式错误和网络错误使用不同策略，Repair 也消耗调用预算。

## 案例四：设置费用上限，但 Provider 不返回 Usage

**现象**：无法计算本次调用费用。

**处理**：Fail Closed，不将未知费用当成 0。

**面试重点**：预算系统必须覆盖“不知道花了多少”的情况，否则费用上限只是展示字段。

## 案例五：运行中收到取消请求

**现象**：任务已经进入模型或检索流程。

**处理**：先持久化 `cancel_requested`，Executor 在下一工作流边界检查。已经发出的远程调用可能仍计费。

**面试重点**：Python 异步任务一般不能安全地在任意指令位置强制终止，协作式取消更可控。

## 案例六：进程在远程调用后崩溃

**现象**：Provider 可能已经执行成功，但结果没有完成本地事务。

**处理**：重启时将遗留任务标记为 `PROCESS_RESTARTED`，不自动重放。

**面试重点**：没有 Provider 级幂等保证时，自动重试可能重复计费。Exactly-once 不能靠一句“重启后重试”实现。

## 案例七：两个 Worker 同时领取任务

**现象**：并发 Claim 同一个 queued task。

**处理**：领取发生在 SQLite `BEGIN IMMEDIATE` 事务中，并使用状态条件 UPDATE；只有一方影响一行并成功。

**面试重点**：测试不只跑单线程，仓库包含并发 Claim 基线。

## 案例八：两个页面同时修改 Review

**现象**：旧页面提交覆盖新页面结果。

**处理**：客户端携带 `expected_version`。版本不匹配返回 409。

**面试重点**：轻量级乐观锁比最后写入覆盖更适合人工审查数据。

## 案例九：安装了两个同名插件 Entry Point

**现象**：授权名称对应多个候选包。

**处理**：拒绝加载全部候选，并返回 Duplicate 错误。

**面试重点**：不能依赖安装顺序静默选择，因为这会破坏可重复性和供应链边界。

## 案例十：数据库来自更新版本程序

**现象**：`PRAGMA user_version` 高于当前支持版本。

**处理**：启动失败，不允许旧程序继续写入。

**面试重点**：向前迁移可以做，未知的向后兼容不能猜测。
