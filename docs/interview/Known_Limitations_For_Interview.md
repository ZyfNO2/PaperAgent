# PaperAgent · 面试中的已知限制与诚实表达（Session 43）

## 当前最重要的真实限制

### 1. Step 6 导出依赖后端 `18181`

- 现状：前端工作台可独立演示，但真正生成导出结果仍依赖后端接口。
- 面试表达：前端会明确提示后端离线，不会伪装成功。

### 2. Demo Case 是固定演示数据

- 现状：`Interview Mode` 里的 Demo Case 用于稳定讲解，不代表实时联网检索。
- 面试表达：这是为了保证现场演示可控，同时仍保留真实工作流结构。

### 3. RAG 更偏可解释轻量实现，不是重型向量检索系统

- 现状：当前强调 query 拆解、候选召回、重排和证据晋升。
- 面试表达：向量库是扩展位，不是当前 MVP 的核心卖点。

### 4. LangGraph runtime 目前只是映射，不是默认执行引擎

- 现状：现在是轻量状态机 + Gate。
- 面试表达：我先把工作流设计成可映射到 StateGraph，而不是为了面试硬接重依赖。

### 5. SubAgent / Multi-Agent 仍是设计预留

- 现状：界面里可以讲解 Supervisor、Router、权限边界，但当前主链路仍是单流程。
- 面试表达：顺序强、Gate 强的选题流程，过早拆多 Agent 收益有限。

### 6. MCP / 工具边界不默认开放写能力

- 现状：深挖里会讲 read-mostly 边界，但不宣称当前默认开放所有写操作。
- 面试表达：这是为了避免“为了展示工具而扩大权限面”。

## Session 43 新增的诚实边界

### 普通模式和面试模式是同一套工作台，不是两套系统

- 好处：避免演示版和真实版脱节
- 边界：面试增强层只负责讲解，不接管业务状态

### implemented / lightweight / design-only 必须明确区分

- implemented：当前已经在 UI 或主链路里实际可演示
- lightweight：有稳定轻量实现，但不是重型生产版本
- design-only：只做架构预留，不能说成已完整落地

## 建议表达模板

### 面试官问“你这个 demo 是真跑的吗？”

可回答：

> 工作台和交互是真的，Demo Case 是固定演示数据。我刻意把这两件事分开讲：一部分用于稳定复现场景，一部分用于说明真实执行边界，比如后端离线时前端会明确提示而不是假成功。

### 面试官问“为什么现在不用 LangGraph / 向量库 / 多 Agent？”

可回答：

> 因为当前目标是把核心工作流做清楚：Step、Gate、Trace、WorkspaceCommand 和导出前检查。重型依赖都保留了映射和扩展位，但没有为了短期好看把项目做重。

### 面试官问“你项目最大的限制是什么？”

可回答：

> 现在最大的真实限制是 Step 6 仍依赖后端接口，以及 RAG 深挖仍偏轻量实现。好处是我能非常清楚地区分哪些是当前可演示能力，哪些是扩展位，不会把设计稿说成已上线能力。


## 应对策略与后续规划

针对上面六条限制的应对策略与后续方向（面试时被追问“你打算怎么解决”可直接用）：

1. Step 6 导出依赖后端 `18181`
   - 应对：前端做硬性 readiness查，后端离线就明确提示，不假成功。
   - 后续：把导出能力下沉为可离线的轻量生成路径，降低对在线后端的强依赖。

2. Demo Case 是固定演示数据
   - 应对：在 UI 上标注来源，区分“稳定演示数据”和“真实检索结果”。
   - 未来：支持接入一次真实在线检索，作为与 Demo Case 对照的演示例。

3. RAG轻量实现
   - 应对：保留 Candidate → Evidence 门控与引用链，向量库留作扩展位。
   - 后续：补 Hybrid Search / Rerank / Eval 闭环，但不在 MVP 主链路强行接入。

4. LangGraph 只做映射不做默认执行
   - 应对：状态机 + Gate 已经覆盖主链路的确认与回退需求。
   - 未来：子流程膨胀时再评估是否接入真实 StateGraph runtime。

5. SubAgent / Multi-Agent 仍是设计预留
   - 应对：当前单流程 + Gate 收益已足够，过早拆多 Agent 会增加路由复杂度。
   - 后续：若子 Agent 数量膨胀，再做层级路由与并行评审。

6. MCP / 工具边界不默认开放写能力
   - 应对：默认 read-mostly，避免为了展示工具而扩大权限面。
   - 生产环境：再按权限审查逐步放开，并配审计日志。

这样“限制 + 应对 + 后续”三段式齐备，能体现对边界的主动管理，而不是只列问题不给方向。
