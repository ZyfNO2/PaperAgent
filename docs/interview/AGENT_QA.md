# PaperAgent Agent 面试问答

## Agent 和普通工作流有什么区别？

PaperAgent 不把单次模型调用称为 Agent。它包含持久化状态、检索工具、结构化决策、失败分支、预算、人工 Review 和事件流。为了可控，决策空间是有限的，而不是开放式无限执行。

## 为什么使用 LangGraph？

研究流程包含规划、检索、证据综合、方法设计和报告等显式节点。LangGraph 让状态转换、节点边界、重试和取消检查可以分别测试。Provider、Store、Clock 和 ID Factory 仍通过服务对象注入，业务逻辑不绑定具体供应商。

## 如何防止无限循环？

- 图结构有限；
- Retry 次数有上限；
- Repair 最多一次；
- 每任务有调用、Token、时间和费用预算；
- 检索迭代有上限；
- Budget 错误不重试。

## 为什么要求结构化输出？

后端需要稳定字段驱动下一节点、持久化和评测。自由文本不能直接作为程序状态。每次响应都经过严格 Pydantic 校验；不合法时最多一次 Repair，仍不合法就失败。

## Schema 正确是否代表结论正确？

不代表。Schema 只证明结构和基本约束正确。科学质量还要验证引用标识、Claim-to-Evidence、证据冲突、证据不足、外部 Holdout 和人工盲审。项目明确区分工程验证与科学质量验证。

## Retry 和 Repair 有什么区别？

Retry 处理网络、限流或服务端暂时失败，逻辑请求不变。Repair 针对已经返回但不符合 Schema 的内容，是一次新的模型调用。两者共享预算但分别记录，避免用 Repair 掩盖首轮质量。

## 如何控制费用？

每个任务持有统一 Budget Ledger。每次物理调用，包括 Retry 和 Repair，都会消耗调用次数、Token、时间和费用预算。如果设置费用上限但无法取得 Usage 或价格，系统直接停止，而不是将未知费用视为 0。

## 如何减少恶意上下文影响？

- 用户输入和检索内容仅作为数据；
- 可调用能力集合固定；
- Provider 配置不进入任务 Payload；
- 输出必须满足受限 Schema；
- 对抗场景独立评测；
- Telemetry 递归脱敏。

这些措施降低风险，但不声称完全消除模型层风险。

## 如何验证引用？

文献 Provider 分离发现和验证。系统记录来源标识、Locator、获取时间、验证状态和内容 Hash。验证失败的候选不能被 Review 为 Accepted。正式发布还需要元数据匹配和 Claim-to-Evidence 指标。

## 为什么需要 Human-in-the-loop？

科研结论需要领域判断，不能仅凭格式正确直接发布。Review 层允许用户接受、拒绝和收藏证据，Export 只导出明确选择的集合，并使用版本号处理并发更新。

## 插件与工具有什么区别？

工具是工作流可调用的受控能力；插件是扩展 PaperAgent 的本地 Python 包。外部插件不会自动加载，必须由操作员在当前命令中精确授权。当前授权插件仍在主进程运行，因此不是隔离环境。

## 怎么评测 Agent？

1. 单元合同：Schema、预算、重试、脱敏；
2. 图级 E2E：正常路径、超时、错误结构、Repair、取消和检索重试；
3. HTTP 垂直链路：Submit、Progress、Review、Export；
4. 真实 Provider Smoke：连接和协议；
5. 开发集：领域内、领域外、证据不足和对抗；
6. 外部 Holdout 与盲审：发布前科学质量证据。
