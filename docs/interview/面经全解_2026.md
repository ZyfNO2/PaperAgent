## 2. 8 主题深度作答

>节结构：标准答（可直接念）/ 2026 点方法串联 / PaperAgent么讲（三档口径 + 文件路径）/ 企业级回答版 / 反问防线（追问 + 应对）/ 3-6 个真实问法配精炼答。

### 2.1 RAG统一设计（含专业数据库 + Hybrid + Rerank + 多层验证 + PaperKG）

**标准答（可直接念，约 400 字）**

RAG 不是"把文档切片塞进向量库再相似度召回"这么简单。生产级 RAG 我会把它拆成四段可插拔管线：QueryPlanner查询解析与多路计划、Retriever粗排召回、Reranker精排、Verifier多层引用验证，最后 Reporter着引用输出。粗排用 Hybrid Search——BM25疏检索负责关键词与专有名词精确匹配，Dense Embedding密检索负责语义同义，两路用 RRF（Reciprocal Rank Fusion，`score=Σ 1/(k+rank_i)`，k=60）融合，不用加性融合因为不同源分数量纲不同会翻车。精排对 top-50选用 CrossEncoder（如 bge-reranker-base，把 query 和每个文档拼一起过模型）排到 top-10，成本和精度的经典 trade-off。验证层我做四层：L1 可链接性（HTTP 200 + metadata整）、L2 来源权威性（peer-reviewed venue / arXiv DOI）、L3 元数据完整性（references / dataset 字段非空）、L4 内容相关性（LLM定但只出 verification_report 不入 supports）。评估闭环用 Recall@K / MRR / Citation Coverage / Evidence Precision 四指标，对标 Ragas 的 faithfulness / context_precision 但自建不引重依赖。跨文档关联用 PaperKG 关系图（cites / extends / contradicts / uses_same_dataset），做 1-hop域查询喂给不同 Gate，对标 GraphRAG部查询但不引 Neo4j、不用 LLM关系。

**2026 点方法串联**

GraphRAG（微软，实体-关系-事件建图 +区摘要，解决跨文档关联推理）+ Agentic RAG（Self-RAG / Corrective RAG，Agent 主导多轮规划、多源路由、答案验证，把一步式检索升级为可自我纠错的检索行为）+ Hybrid Search（ES 的 BM25 + kNN 是工业默认，开源侧 Chroma/Qdrant/Weaviate 同构）+ RRF合（Cormack 2009，只用 rank 对尺度不敏感）+ CrossEncoder Rerank（bge-reranker / Cohere Rerank / Jina）+ 评估 Ragas/DeepEval + 向量库 Qdrant（Rust、gRPC、过滤强、生产级）/ Chroma（开发友好嵌入式）/ pgvector（企业已有 PG 最省）/ Milvus（十亿级分布式）。

**PaperAgent么讲（三档口径 + 文件路径）**

- **implemented**：QueryPlan 7索层、多源召回（OpenAlex/arXiv/GitHub/HuggingFace，Session 14）、CandidateResource 结构、EvidenceRef 单层 URL Verified（arxiv/github/huggingface/kaggle/generic_url，confidence 0.0-0.85）、RagEvalReport baseline。代码在 `apps/api/app/services/rag_pipeline.py` / `apps/api/app/services/rag_evaluator.py` / `apps/api/app/schemas_rag_eval.py`（5 因子可调：keyword_match 0.35 / url_verified 0.20 / reproducibility 0.25 / type_coverage 0.10 / recency 0.10，URL 未验证 *0.4罚），测试 `test_session34_rag_pipeline_eval.py`（25 个）。讲法引 `docs/interview/RAG_Design_Explainer.md`。
- **lightweight**：当前是 mock BM25（关键词 Jaccard）+ mock Dense（title+abstract token overlap × year weight）+ 真 RRF合 + 多因子 Rerank，向量库为占位。5存储：文件指纹库（SQLite SHA256等）/ Dense 向量库 / BM25排索引（JSON）/ 图片资源库 / Trace 日志（JSONL）。讲法："我没把 RAG 写成一个检索函数，而是 QueryPlanner+Retriever+Reranker+Verifier+Reporter 五段可插拔接口，对标 Haystack Pipeline，当前 lightweight 用 mock 占位、向量库留接口"。
- **design-only**：CrossEncoder Rerank（bge-reranker-base）、Qdrant 适配层（本地 FAISS/numpy 占位）、多层验证链 L1-L4 + `verification_report.json`、PaperKG 多论文关系图（`LinkwiseContext` + `data/paperkg.json` JSON + 内存子图，1-hop域查询，label-propagation区发现，无 LLM区摘要）。讲法引 `Plan/design/RAG_ProfessionalDB_Upgrade.md` + `docs/interview/AutoResearchClaw_对标与小型化移植.md` §14。守不变式：LLM 只判 L4 相关性，不写 evidence supports。

**企业级回答版**

"对标 Haystack Pipeline / LangChain RunnablePassthrough / LlamaIndex QueryEngine 的 Retriever/Reranker/Verifier/Reporter 四段划分；Hybrid Search 对标 ES 的 BM25+kNN，RRF 是 Cormack 2009业默认；CrossEncoder Rerank 对标 Cohere Rerank / Jina Reranker；多层验证对标 Great Expectations / dbt tests / OpenLineage 数据血缘；熔断对标 Resilience4j / Sentinel 的 closed/half_open/open；评估对标 Ragas（faithfulness / context_precision / answer_relevancy）。我没引 Ragas 是因为 MVP 不加重依赖，自建四指标口径与 Ragas 对齐。"

**反问防线**

-问 1："RRF 为什么不用加权平均？" → 答：稀疏和密集分数尺度不一致，加权需要先归一化引入额外参数；RRF 只用 rank 对尺度不敏感，可自然扩展到 N 个 retriever。k=60 是经验值。
-问 2："CrossEncoder双塔慢 100为什么还要用？" → 答：只对 top-K（K=20）用，不是全量；双塔负责粗排快召回、CrossEncoder责精排抠精度，这是成本精度经典分层。
-问 3："为什么不直接训练一个 cross-encoder rerank？" → 答：训练成本高（relevance 定义因校而异标注贵）、推理慢；多因子加权可解释可调，面试时能讲清每个因子来源。 design-only扩展位。
-问 4："你的向量库为什么不用 Milvus？" → 答：数据量级在百万 chunk别，倒排索引内存 1-2GB得住；本地化部署不想引 Milvus/ES 重依赖；千万级以上才上分布式。接口隔离不锁死 Qdrant/FAISS。

**真实问法配精炼答**

1. "你的 RAG统都存了哪些数据？怎么存的？" → 5存储：文件指纹库（SQLite重）/ Dense 向量库（Chroma HNSW O(logN)）/ BM25排索引（JSON，O(1)找 + O(K)序）/ 图片资源库 / Trace 日志（JSONL）。
2. "硬切分有哪些弊端？" →坏语义边界（概念步骤被切开）、边界信息丢失（版本号/接口名/错误码落交界）、embedding果变差（平均语义、上下文断裂）。改用语义优先 +度约束 + 10-20% overlap + 评估验证。
3. "ChunkSize么选？" → 4法：切割策略 → embedding 最佳区间（BGE 100-400 token，ada-002 100-800）→ 文档自然粒度（FAQ 100-300 字 /术文档 300-600 / 法规 500-1000）→ LLM口预算（top-5/10 chunk × 500-1000 token）。
4. "为什么不用 LlamaIndex？" → 设计目标不同：LlamaIndex 广度优先（200+成），自建深度优先（每个环节可控可监控可调优）；LlamaIndex 依赖链太重（SQLAlchemy/networkx/nltk/tiktoken 等 200+ 传递依赖），启动慢内存大版本冲突。
5. "RAG 传统方案有什么不足？新方向？" → 传统把知识"压扁"成一维 chunk，丢失版面结构、跨文档关系。新方向：GraphRAG（实体关系图谱 +区摘要）解决知识表达局限；Agentic RAG（多轮规划 +由 +证）解决检索过程被动。两者结合是趋势。
6. "怎么保证召回内容完整可用？" → 三层：知识进入时完整（按结构切 + 保留 metadata + 图 caption）+索时召回全排得准（Hybrid + RRF + Rerank）+ 生成时可追溯可评估（citation + Ragas/DeepEval标）。

### 2.2 多 Agent架构设计（5表 + Supervisor膨胀最优解）

**标准答（可直接念，约 450 字）**

多 Agent 不是"为多而多"。设计前先判断任务有没有并行分工价值——强顺序、长链路推理、工具密集、单 Agent baseline 已经高的场景，盲目拆多 Agent 反而引入协调成本、通信开销和误差传播。架构选型有 5 种：SAS 单体（O(T) 顺序，0 通信，最稳，baseline>45% 时加协调收益递减为负）、MAS立并行（O(n) 并行 +合/投票，多视角子任务）、中心化 Supervisor（Supervisor颈，强规划分发）、去中心化 Peer（通信开销大但容错强，动态环境/检索路由）、混合 Hierarchical（层级 + Peer，通用推荐）。子 Agent胀到 20-50 时 Supervisor力大，最优解是组合拳：第一层 Hierarchical 降扇出（每层扇出 b≤5-7，b=5 d=3 = 125量，胜单层线性），对标组织学/B+树；第二层分类用多 Agent票 +信度门（同层 N 个轻量分类器 majority/Borda，低置信走人工/回退）；第三层并行 + 异步降延迟（同层并行跨层串行，LangGraph parallel branch，投票别改串行）；第四层处理速度 vs错一致性取舍（Barrier 合并快慢差、显式熔断 closed/half_open/open + negative cache + 降级 heuristic、用 Saga 最终一致而非强一致 2PC，因为 Agent 是"给建议人留门"不是金融交易）。量纲上中心化复杂度 O(n×k×r) n性涨，Hierarchical度 log_b(n) 主席度降但总 joint可接受，投票并发 wall-clock = max(同层) n 串行。

**2026 点方法串联**

Google《Towards a Science of Scaling Agent Systems》经验阈值（单 Agent baseline 45% 时加协调收益递减为负）+ Hierarchical出 b/d 数学（b^d量）+ LangGraph StateGraph + checkpoint + time travel（节点回放重跑）+ CrewAI/AutoGen色协作 + Resilience4j 显式熔断 + Saga偿事务（最终一致）+ Airflow/Temporal作流编排 +投票 majority/Borda +信度门。

**PaperAgent么讲**

- **implemented**：当前 PaperAgent = SAS 单体 + 4 Gate（keyword gate / candidate gate / promotion gate / readiness gate），顺序选题流程（题目→关键词拆解→检索计划→候选证据→证据晋升→可行性判断→选题报告→导出前检查）。7 Agent Role schema 已定义（SupervisorAgent / KeywordAgent / RetrievalAgent / VerificationAgent / FeasibilityAgent / ProposalAgent / ReviewAgent），静态路由表 `TASK_TYPE_TO_ROLE`（静态胜 LLM由省 1调用、可测、可审计），CostBudget（max_agent_count=8 / max_llm_calls=20 / max_parallel_tasks=3 / max_rounds=5 + fallback_to_single_agent / early_stop_on_gate_blocked），`tally_votes()`单多数。代码 `apps/api/app/services/agent_router.py` + `apps/api/app/schemas_agent_plan.py`，测试 `test_session37_multi_agent_design.py`（28 个）。讲法引 `docs/interview/MultiAgent_Expansion_Design.md` + `docs/interview/Deep_Dive_QA_Agent.md`。
- **lightweight**：多源并行召回已是隐式 MAS（OpenAlex/arXiv/GitHub/HF 并行 + RRF合）。
- **design-only**：真多 Agent Supervisor 编排（当前 `agent_router.py` mock 数据，未接真 LLM）、Hierarchical 多层 + 并行投票（confidence 门）、LangGraph射（不执行，checkpoint 可回放对照）、显式 CircuitBreaker。讲法引 `Plan/design/MultiAgent_Architecture_Evolution.md` + `docs/interview/AutoResearchClaw_对标与小型化移植.md` §2.7 / §3.1（MetaClaw）/ §3.3（熔断）。

**企业级回答版**

"对标 CrewAI Supervisor + LangGraph checkpoint/time travel + Resilience4j断，小型化取舍为'自写以讲清原理代替重依赖'。Saga偿事务对应 Agent败的回滚；LangGraph checkpoint 对应 PaperAgent Trace 已天然支持的回放到某步重跑；Airflow/Temporal 对应多 Agent 顺序/并行 seat。不用强一致 2PC 因为 Agent 是给建议人留门不是金融交易。"

**反问防线**

-问 1："你为什么现在不用多 Agent？" → 答：PaperAgent 是顺序选题推理，一步错步步错，过早拆多 Agent 反而加汪路由复杂度。对标 Google 论文 45%值——单 Agent baseline高时加协调收益递减为负。3 个触发条件（检索源≥5 / 评分模型≥3 /板类型≥5）满足才拆。
-问 2："Supervisor 下子 Agent胀到 20-50么办？" → 答：组合拳不是单点——Hierarchical 降扇出（b=5 d=3 = 125量）+ 同层投票降分类错率 + 并行降延迟 + 显式熔断 + Saga 最终一致。
-问 3："投票性能开销大怎么办？" → 答：同层并行跨层串行，wall-clock = max(同层) n 串行；用 LangGraph parallel branch 别改串行。
-问 4："第一层分错后面全错怎么办？" → 答：投票 +信度门，低置信走人工/回退路径；不是强一致，接受最终一致预估。

**真实问法配精炼答**

1. "多 Agent 时要怎么设计这个系统？" → 先判断是否一定要用多 Agent（决策树：动态环境→去中心化；顺序长链推理→SAS+Gate；分层分发→Hierarchical+既有 Skills；多视角并行→MAS+加权投票）；再看 5表选型。
2. "你的项目用什么多 Agent架构？" → SAS + 4 Gate，因为顺序选题一步错步步错；多 Agent design-only Supervisor + Retriever/Verifier/Reviewer。
3. "子 Agent胀到 20 个/50 个怎么优化？" → Hierarchical +投票 + 并行 +断 + Saga（见上_standard答）。
4. "多 Agent作遇到的问题？" → 四个问题：架构选型（是否值得拆）、通信成本（粒度+轮数+结构化结果）、状态管理（私有上下文+共享结果区）、错误传播（减少放大）。

### 2.3 量化与推理加速（诚实口径：PaperAgent API 侧消费不做本地量化）

**标准答（可直接念，约 400 字）**

推理加速我从两条线讲：模型层量化 +统层推理框架。量化方法上，INT8/INT4 是权重位宽，越低越省显存越快但越掉点；AWQ（Activation-aware Weight Quantization）看激活值哪些通道重要重点保精度，GPU用；GPTQ于二阶 Hessian 信息后训练量化逐层校准，GPU；GGUF Q4_K_M 是 llama.cpp/CPU 分块混合精度（重要层高 bit、次要层低 bit，全局 4-bit 平均），移动/边缘首选；BF16练数值稳、FP8 H100理用。压缩技术族谱还有蒸馏（训小模型模仿大模型 logits）、剪枝（结构化剪通道<非结构化稀疏）、二值化（1-bit究向）。系统层核心是让 GPU 别闲着：KV-Cache存 Transformer 的 K/V免重算历史 token（长序列省平方级开销）、PagedAttention（vLLM）分页管 KV 解碎片对标 OS 页表、Continuous Batching态拼 batch、Prefix Cache/Prompt Caching用公共 system prompt 的 KV、Speculative Decoding 小模型猜 token + 大模型并行校验、Chunked Prefill 长 prompt 分片不阻塞 decode。推理框架：vLLM（PagedAttention + continuous batching吐王）、SGLang（结构化输出 + RadixAttention缀树复用 KV，工具调用/JSON 快）、llama.cpp（CPU/边缘 GGUF）、TensorRT-LLM（NVIDIA选）。

**2026 点方法串联**

Q4_K_M合精度（Attention 5-bit / FFN 4-bit /出 6-bit）+ AWQ + GPTQ + BF16/FP8 + KV-Cache + PagedAttention（vLLM）+ Continuous Batching + Prefix Cache + Speculative Decoding + Chunked Prefill + vLLM/SGLang/llama.cpp/TensorRT-LLM + Resilience4j（推理服务降级）。

**PaperAgent么讲（诚实口径，关键）**

**先诚实说**：PaperAgent 是 API 侧消费（调 Minimax外部 LLM API），不做本地量化/推理。这条必须先说，不冒充。依据 `Plan/interview_aid/Quant_Inference_SpeedCard.md` §5。

**如果被追"那本地部署的话你会怎么做"**（design-only，不落实现）：

> "PaperAgent 如果在 CPU（边界设备场景），我会选 Q4_K_M合精度——Attention 保 5-bit、FFN 4-bit、输出层 6-bit，保关键层精度。用 llama.cpp GGUF 本地推理，系统层用 prefix cache用同一题目流的公共 prompt（system + RAG context），降低重复 prefill 成本。如果上 GPU 服务，换 vLLM + AWQ。这整套对应企业级'推理服务降本'讲法。"

收尾诚实一句："PaperAgent MVP 先把 Agent作流做清楚，量化/推理服务是部署扩展位，不是当前核心卖点。"——守住 Technical_Highlights 三档口径。

**企业级回答版**

"量化主题面试官通常理解不深、问概念为主（小红书场博主明确指出这一点），所以给关键词 + 一层机制就够，别被拽进 GPTQ Hessian 公式细节泥潭。如果对方深问，诚实说'这块我不是专家，但我清楚量化的应用层取舍和适用场景'——分比硬装高。"

**反问防线**

-问 1："量化会降低精度，什么影响？" →说：离群激活通道误差放大→精度降；层说：输出分布漂移→长尾/罕见 token 量降。可校准 +合精度缓。
-问 2："量化和 KV-Cache 什么关系？" → 两种不同技术可结合：量化降权重/激活位宽省显存，KV-Cache免重算历史。量化也可作用在 KV-Cache 上（KV quant）双省。别只说"没关系"完事。
-问 3："Q4 和 Q8 多大差别？" →一半显存，质量 Q4 约 Q8 的 95% 上下、明显优于 Q2。给数值感：显存 -50%、质量 -low 个位数%。
-问 4："为什么大模型不直接 FP8/INT4 全用？" →练稳用 BF16，FP8 H100；尾部 token 多保精度。

**真实问法配精炼答**

1. "你们项目是怎么做推理加速的？" → 诚实：API 侧消费不做本地量化；如果做给 Q4_K_M + llama.cpp + prefix cache 方案（design-only）。
2. "量化的好处是什么，缺点是什么？" →处：推理快、成本低、能跑更小卡；缺点：精度损失（尤其离群激活通道）、输出分布漂移。
3. "量化在 KV-Cache 中起什么作用？" → 不同技术可结合，KV quant 双省。
4. "部署一个大模型服务器真正要关心什么？" → 不是"能不能跑起来"，而是性能（QPS/并发/首 token迟/p95p99）、资源成本（显存/GPU 利用率/KV cache/batch/量化）、稳定性（超时/限流/熔断/降级/重试/健康检查/多副本）、可观测（耗时/错误率/GPU 利用率/tokens/s/OOM/版本管理/灰度/回滚）。
5. "你了解哪些推理框架？" → vLLM（PagedAttention + continuous batching吐王）/ SGLang（结构化输出 + RadixAttention）/ llama.cpp（CPU GGUF）/ TensorRT-LLM。

### 2.4 Agent记忆（四层架构 +压缩 + 写入 + Replay）

**标准答（可直接念，约 450 字）**

Agent忆系统不是"存个日志"。成熟的做法是分层——ClaudeCode 的四层记忆是 2026 年 3 月源码泄露后的必考参考实现：指令文件（CLAUDE.md 族，人写的硬性规则，五层企业级/用户级/项目级/规则级/本地级，冲突时叠加不覆盖、靠模型注意力机制+加载顺序软优先级）、对话上下文（messages[] 内存数组，短期工作记忆）、Session Memory（summary.md 当前会话结构化笔记）、Memdir久记忆（MEMORY.md引 + topic_*.md 主题文件，YAML frontmatter name/description/type，跨会话）。记忆系统旁边还有一个 Transcript 会话账本（JSONL），保存完整事实（用户输入/助手回复/工具调用/工具结果/compact boundary/错误恢复痕迹），用于恢复、回放、审计、续接。压缩策略 CC 是五层（Tool Budget / Micro-compact Gentle→Aggressive / Auto-compact心层 / Block / Reactive），Auto-compact发阈值以 200K口为例是 167K token，先轻量 Session Memory Compact（保最近 10-40K，无需额外 LLM用），不行再 Full LLM Compact（输出 20K，9 个固定段落）。写入机制三种：extractMemories（每轮对话后提取新记忆入持久存储）、Session Memory（会话过程中维护工作笔记）、AutoDream（距上次整合≥24h 且≥5 会话且拿到 PID 文件锁触发，4段 Orient→Gather→Consolidate→Prune，整合期间只读访问只能写记忆目录）。关键设计哲学：代码用 grep/glob时搜索不写入 Memory（代码会变但记忆不自动更新，过时代码记忆变危险误导），只有人的偏好和判断才持久化；MEMORY.md引≤200 行保持简洁。压缩是"同步"不是异步（异步会引入读写并发问题），JSONL 比 Markdown 更适合机器读取和追加写入。

**2026 点方法串联**

ClaudeCode 四层记忆 + Transcript JSONL + 五层压缩（Auto-compact 核心）+ 三种写入（extractMemories / Session Memory / AutoDream 4段）+ LangGraph Persistence（Graph→Super-step→Checkpoint→Thread→StateSnapshot，Time Travel点回退）+ OpenClaw 四层记忆对比 + 代码实时搜索/偏好持久化哲学。

**PaperAgent么讲**

- **implemented**：PaperAgent 四层记忆——ShortContext（浏览器运行时 Step Deck runState）→ Transcript（RunEvent JSONL `events.jsonl`，可 replay）→ ProjectMemory（ProjectMemorySnapshot 项目级摘要，跨 session 保留，token 重字段从 transcript离）→ EvidenceMemory（EvidenceRef + URLVerified + Promotion，不可压缩/不可覆盖，最高可信）。压缩保留 6 类 critical 事件 100%（user_patch / gate / evidence_promotion / url_verified / readiness_check / llm_call），非 critical is_compressed=True。Replay 三步：加载 snapshot（cold start）→ 加载 transcript events（from_seq 之后）→ 合并 step_states 重建前端。`replay_source` 字段（snapshot / transcript / both）告诉前端数据来源。代码 `apps/api/app/services/project_memory.py` + `apps/api/app/schemas_memory.py`，测试 `test_session35_agent_memory_replay.py`（14 个）。讲法引 `docs/interview/Agent_Memory_Explainer.md` + `docs/interview/Deep_Dive_QA_Memory.md`。
- **lightweight**：Snapshot 在内存（未持久 SQLite），JSONL 性能磁盘级。压缩阈值默认 200，保留最近 N 个。
- **design-only**：Snapshot久 SQLite、EvidenceMemory 远程 DB、多用户冲突解决（last-write-wins + Trace）、自动 snapshot 生成、跨 project 共享 ground truth。讲法诚实说："当前 Snapshot 在内存、JSONL盘级，这是已知边界（`Known_Limitations_For_Interview.md` 第 3/5 条）"。

**企业级回答版**

"对标 ClaudeCode 四层记忆 + Transcript 双层（语义资产 vs 事实账本）；LangGraph Persistence 的 Graph→Super-step→Checkpoint→Thread→StateSnapshot 对应 PaperAgent 的 ProjectMemorySnapshot + RunEvent replay；AutoDream 的 Orient→Gather→Consolidate→Prune 对应 ProjectMemory缩时的 critical 保留 + Prune。CC 五层压缩（Auto-compact 核心）对应 PaperAgent 的阈值压缩 + snapshot离 token 重字段。守'代码实时搜索/偏好持久化'哲学——PaperAgent 不持久化代码事实（会变），只持久化用户 patch/gate策/evidence。"

**反问防线**

### 2.5幻觉抑制（H-Neurons + RAG + RLHF + CoT +投票）

**标准答（可直接念，约 400 字）**

大模型幻觉的本质不是"故意撒谎"，而是训练方式和训练数据共同塑造的生成倾向——底层是概率预测器不是知识库，内部没有"事实文件柜"只有词与词的概率地图，生成一句话等于沿着最可能的路径一步步走，从设计上就没有"真/假"概念只有"像不像下一句"。因果链可以拆成四层：根因（底层机制 =率预测器，开口就可能编，错误随长度累积）、燃料（训练数据残缺扭曲——截止日之后事实无路径、冷门知识路径模糊、数据污染、过拟合）、扳机（训练偏好让它"尽量给答案"——预训练学到"顺着说"惯性，SFT/RLHF/DPO"给可用答案"倾向）、电路（清华 H-Neurons论文：几十亿参数中不到十万分之一的神经元与幻觉强相关，放大 H-Neurons模型接受错误前提/附和用户/越狱，驱动的是"过度服从"而非"错误记忆"，且和语言流畅性深度纠缠无法简单删除）。工程策略是"绕过它、监测它、约束它、复核它"：RAG（把回答锚定到可信知识库，让模型不独自承担事实真实性）、RLHF/DPO（显式奖励"承认低置信度"行为）、CoT/Self-Ask（多轮验证反思，分更多算力强迫展开自我核查）、融合投票（多模型/多 Agent票或加权，群体智慧稀释个体偏差）、外部工具/Web Search（让模型先查证再生成）、事实核查与用户侧验证（建立事实校核机制而非从模型内部消除）、H-Neurons测（监测活跃度异常飙升给"可能在撒谎"信号，盖面抑制预警而不是粗暴切除）。

**2026 点方法串联**

清华 H-Neurons论文（不到十万分之一的神经元驱动"过度服从"）+ RAG锚定 + RLHF/DPO奖励低置信度诚实 + CoT/Self-Ask反思 + Self-Consistency多样本投票 + Corrective RAG/Self-RAG我纠错 + LLM-as-judge（守"只判相关性不入 supports"不变式）+ Great Expectations数据质量校验。

**PaperAgent么讲**

- **implemented**：三层幻觉抑制——RAG约束（候选列表有界，不让模型凭空生成）+ Gate检查（URL 必须 Verified能晋升 Evidence）+ Trace审计（所有生成可追溯，user_patch/gate/evidence_promotion 全保留）。PromptProtocol + WorkspaceCommand览确认（action whitelist + 7 forbidden patterns，13 whitelist components、5 whitelist actions）。Candidate→Evidence升是软删除留 Trace + URL Verified + readiness hard block，LLM只给建议不直接写 evidence。讲法引 `docs/interview/AutoResearchClaw_对标与小型化移植.md` §2.3（数据血缘）+ §2.4（PromptProtocol）+ 对应 `docs/interview/Deep_Dive_QA_Agent.md` Q19。
- **lightweight**：URL Verified 单层验证（arxiv/github/huggingface/kaggle/generic_url，confidence 0.0-0.85）。RAG fail case 自动检测（no_dataset/no_repo/url_unverified/low_relevance/type_imbalance）。
- **design-only**：多层验证链 L1-L4（L1 可链接性 / L2 来源权威性 peer-reviewed venue / L3 元数据完整性 / L4 内容相关性 LLM判但只出 verification_report 不入 supports）+ `verification_report.json`。讲法引 `Plan/design/RAG_ProfessionalDB_Upgrade.md` §5 + `docs/interview/AutoResearchClaw_对标与小型化移植.md` §3.2。守不变式：LLM只判 L4 不写 evidence supports，这是 evidence则的核心。

**企业级回答版**

"对标 Great Expectations / dbt tests / OpenLineage质量校验——多层验证链就是数据质量 Gate。H-Neurons盖面抑制预警对标'监测而非切除'。RAG锚定对标'让模型不独自承担事实真实性'。守'LLM 只判相关性不写 evidence'不变式——LLM 的输出是 verification_report 不是 supports，这是防幻觉从设计上而非 prompt上。"

**反问防线**

-问 1："大模型为什么有幻觉？" →四层因果链（根因概率预测器 +料数据残缺 +机训练偏好 + 电路 H-Neurons过度服从）。
-问 2："幻觉具体原因在哪里？" →根因层（无事实数据库只有概率地形）+燃料层（截止日/稀缺/污染/过拟合）+扳机层（预训练惯性 + SFT/RLHF/DPO"给答案"）。
-问 3："怎么减少幻觉？" →六类：外部工具 / RAG锚定 / RLHF/DPO奖励诚实 / CoT+Self-Ask反思 /合投票 / 事实核查 + H-Neurons测。PaperAgent 体是 RAG约束 + Gate检查 + Trace审计三层 + design-only L1-L4 多层验证。
-问 4："H-Neurons能不能直接删除解决幻觉？" → 不能。论文明确不建议删除，因为这条神经元和语言流畅性深度纠缠，更现实方向是盖面抑制预警监测活跃度，而不是粗暴切除。

**真实问法配精炼答**

1. "模型会产生幻觉吗？" → 会，本质是概率预测器 +料 +机 + 电路四层。
2. "怎么减少 Agent 的幻觉？" → RAG锚定 + Gate检查 + Trace审计（PaperAgent implemented）+ 多层验证 L1-L4（design-only）。
3. "RAG 能完全消除幻觉吗？" → 不能完全消除但显著缓解——让模型不独自承担事实真实性，锚定到可追溯的检索结果。
4. "你怎么验证模型答的是对的？" → Citation（source_file/page/chunk_id/score）+ URL Verified + faithfulness评估（对标 Ragas）+ LLM-as-judge守只判相关性不入 supports。

### 2.6 MCP vs Function Calling（标准协议 vs模型调用工具格式）

**标准答（可直接念，约 350 字）**

MCP（Model Context Protocol）是 Anthropic出的标准 JSON-RPC，定位是"Agent具之间的标准协议"——定义 Server / Host / Client 三角，Server露 tools / resources / prompts 三类能力，Host入，Client用，传输支持 stdio / SSE / HTTP。Function Calling 是"模型到工具的调用格式"（OpenAI / Anthropic），模型返回结构化的 `tool_calls` 字段（含 function_name 和 arguments），应用框架读取后执行。两者不是替代关系：MCP 是传输/协议层标准，Function Calling 是模型输出层契约；MCP Server 内部可以被 Function Calling用。本地 Server（stdio，进程间通信，低延迟，同机）vs Server（HTTP/SSE，跨网络，需鉴权，可分布式）的权衡：本地适合隐私敏感/低延迟，远端适合多用户共享/可扩展。Function Calling训练出来的方式：SFT阶段用带 tool_calls构化数据训练，让模型学会在合适时机返回结构化调用而非纯文本；工具调用失败要兜底——解析失败→ fallback prompt、参数错误→校验重试、工具超时→降级 heuristic。2025 年 3 月 OpenAI 出 Response API，把"一次请求里的智能体循环"写进端点契约——状态、工具、事件流、结构化输出纳入服务端，对标 Chat Completions 的过渡，量化智能体能力、降低工程成本。

**2026 点方法串联**

MCP（tools/resources/prompts + stdio/SSE/HTTP）+ Function Calling（tool_calls JSON约）+ OpenAI Response API（2025.03，服务端有状态智能体循环，Chat Completions→Responses过渡）+ A2A议（Google，Agent2Agent，Agent Card / Tasks / Messaging，Agent之间的标准对标 MCP 是 Agent具）+ SKILL（ClaudeCode，SKILL.md + YAML frontmatter，能力发现 + 发）。

**PaperAgent么讲**

- **implemented**：PaperAgent露 4 个 MCP具——`search_topic_evidence` / `get_candidate_resources` / `get_project_trace` / `check_export_readiness`。禁止暴露的具：`promote_candidate_to_evidence` / `generate_proposal_draft` / `delete_project` / `write_file` / `shell_exec`（需用户显式确认）。3权限：whelist（manifest）+ blacklist（`FORBIDDEN_TOOLS`）+ Gate前置条件（`ToolPermission`型 `requires_keyword_gate` / `requires_final_package` / `read_only` / `writes_trace`）。每次工具调用写 Trace（含禁止拒绝），`sanitize_trace_data`对路径脱敏。错误码 `forbidden_tool` / `permission_denied` / `missing_dependency` / `internal_error`（业务错误 HTTP 200 + success=false + error.code，不用 HTTPException）。代码 `apps/api/app/mcp/server.py` + `tools.py` + `permissions.py` + `schemas_mcp.py` + `api/v1/mcp.py`，测试 `test_session36_mcp_tools.py`（19 个）+ `test_one_topic_session36_mcp.py`（6 Playwright）。讲法引 `docs/interview/MCP_FunctionCalling_Explainer.md` + `docs/interview/Deep_Dive_QA_MCP.md`。
- **lightweight**：当前传输是 HTTP（不是真 MCP SDK 的 stdio/sse），这是已知边界。
- **design-only**：stdio/sse 传输、Resource露、Sampling（反 LLM用）、Rate Limiting / billing、跨 project 共享、stats dashboard。A2A "我当前主链路单 Agent+Gate不需要，多 Agent进时会评估 A2A 作为 Agent间标准协议"。

**企业级回答版**

"MCP 对标 LangChain Tools / Anthropic Tool Use / OpenAI Function Calling，关键差异是安全 +计写进协议层——whitelist/blacklist/Gate 三层权限 + Trace 全量审计 +径脱敏 + 业务错误与传输错误分离。设计哲学三条：最小暴露、Gate不可绕过、审计可追溯。默认拒绝显式允许（default-deny, explicit-allow）。"

**反问防线**

-问 1："MCP 和 Function Calling 什么区别？" → MCP 是传输/协议层标准 JSON-RPC（Server/Host/Client + tools/resources/prompts + stdio/SSE/HTTP），Function Calling 是模型输出层契约（tool_calls JSON）。不是替代，MCP Server 内部可被 Function Calling。
-问 2："MCP 本地 Server 和远端 Server 什么区别？" → 本地 stdio 进程间通信低延迟同机隐私敏感；远端 HTTP/SSE网络需鉴权可分布式多用户共享。
-问 3："你的 MCP具怎么防越权？" → 三层：whitelist（manifest须在）+ blacklist（FORBIDDEN_TOOLS 远拒绝）+ Gate前置（requires_keyword_gate / requires_final_package）。写操作 WorkspaceCommand览确认。

-问 5："工具调用失败怎么办？" →底：解析失败→ fallback prompt、参数错误→校验重试、工具超时→降级 heuristic + stale。

**真实问法配精炼答**

### 2.7模型选型与降级（三段论 + 成本监控 + model-agnostic）

**标准答（可直接念，约 350 字）**

模型选型我用三段论：选了什么模型 → 为什么选这个模型 → 有什么具体指标/实验支撑。选型维度看任务类型（理解 vs 成成 vs具调用 vs 代码）、上下文长度、成本（token 单价 +用频次）、延迟（首 token /体生成）、私有化需求（API vs地部署）、工具调用能力（是否原生支持 Function Calling）。降级策略守"input-prefer model-agnostic"——输入优先、模型无关：auto（强模型走 LLM）/ llm（弱模型走轻量 LLM）/ heuristic（兜底启发式），三条路径都跑得通，LLM挂了服务不能挂。CostBudget维硬限：max_agent_count / max_llm_calls / max_parallel_tasks / max_rounds，超过自动降级单进程。还有一个加分项：Agent成本监控——做项目时设置 Token 使用量统计，大概知道用了什么模型、费了多少 Token。强烈推荐做小型实验选型，用真实业务 query 集（gold set）跑 Recall@K / MRR / 任务成功率，数据驱动而非拍脑袋。BERT vs GPT 是经典对比——BERT编码器架构双向上下文遮盖训练擅长理解（分类/问答），GPT码器自回归左到右生成擅长生成（写作/对话），简单说 BERT长"理解" GPT长"生成"。

**2026 点方法串联**

模型选型三段论 + CostBudget维（agent/llm/parallel/rounds）+ input-prefer model-agnostic 降级（auto/llm/heuristic）+ LiteLLM（LLM Gateway 一接口多 provider）+ ACP（Arbitrary-agent backend concept）+ BERT vs GPT（编码器双向 vs码器自回归）+ Agent成本监控（Token计）+ gold set 小型实验。

**PaperAgent么讲**

- **implemented**：PaperAgent外部 LLM API（Minimax 等），降级策略 auto/llm/heuristic 三路径都跑得通，LLM挂了走 heuristic keyword decomposition。CostBudget（max_agent_count=8 / max_llm_calls=20 / max_parallel_tasks=3 / max_rounds=5 + fallback_to_single_agent + early_stop_on_gate_blocked）。LLM一入口 `llm_call(step_key, prompt, **kwargs)` + JSON mode output + heuristic fallback。静态路由表不学 LLM由省 1调用。讲法引 `docs/interview/MultiAgent_Expansion_Design.md`（CostBudget）+ `docs/interview/AutoResearchClaw_对标与小型化移植.md` §2.8（input-prefer model-agnostic）。

- **design-only**：真多模型投票（tally_votes 当前只 schema + mock）、LiteLLM Gateway 适配层、跨 project 共享 ground truth、A/B试不同 RRF k值。

**企业级回答版**

"对标 LiteLLM（一接口多 provider）+ ACP arbitrary-agent backend concept——PaperAgent 的 input-prefer model-agnostic 是'输入优先模型无关'，不强绑定单个 provider。CostBudget维对标 Airflow/Temporal源配额。Agent成本监控对标 Langfuse / promptfoo token计。模型选型三段论是'选了什么 + 为什么 + 什么指标支撑'，第三段必须给数据不能空谈。"

**反问防线**

-问 4："模型挂了怎么办？" → heuristic keyword decomposition底，服务不挂。守"LLM径必须配 heuristic fallback"不变式。

### 2.8 SFT 与微调（LoRA + Prompt Tuning + RLHF + 什么项目该微调）

**标准答（可直接念，约 400 字）**

SFT（Supervised Fine-Tuning）方案主要有：全参数微调（更新所有权重，成本高效果稳但易过拟合）、LoRA（Low-Rank Adaptation，冻结原权重 + 注入低秩矩阵 A/B，只训少量参数，参数量 0.1-1%，显存省 3倍+，适合资源受限）、Prompt Tuning（只训可学习的 prompt 向量，参数更少但能力上限低，适合大模型轻量适配）、Prefix Tuning（Prompt Tuning体加任务前缀）。LoRA常见参数：rank（r，低秩矩阵维度，8-64用）、alpha（缩放系数，控制 LoRA出权重）、target_modules（作用层，常见 q_proj/v_proj）。SFT 和 RLHF 区别：SFT 是监督学习（人工标注的 input-output 对），RLHF 是强化学习（人类偏好反馈训练 reward model + PPO 优化，让模型 align 类偏好）；DPO 是 RLHF化版（直接偏好优化，不需要 reward model，更稳定）。数据来源：开源数据集（HuggingFace / Anthropic HH-RLHF / OpenAssistant）+ 人工标注 + LLM助生成（人为干预检查）+实用户日志筛选。怎么判断一批数据值不值得训练：数据质量（准确性/多样性/无噪声）、数据量（是否够覆盖任务分布）、与目标任务对齐度（分布匹配）、标注一致性。什么项目适合微调 vs只做 RAG：模型需要领域知识内化（高频私域术语）→；需要最新事实/可溯源引用 → RAG；两者结合最优（微调让模型懂领域语言，RAG供事实）。DeepSeek R1方法（RL + 冷启动 SFT +理链蒸馏）是 2025练方法八股常考点。

**2026 点方法串联**

LoRA（r/alpha/target_modules）+ Prompt Tuning + Prefix Tuning + QLoRA（量化 + LoRA + CPU offload，4-bit座 + LoRA练，显存再降）+ Peft + RLHF（reward model + PPO）+ DPO（直接偏好优化，无 reward model）+ DeepSeek R1（RL + SFT +理链蒸馏）+ 数据来源四法（开源/标注/LLM助/日志筛选）+ LLaMAFactory / OpenRLHF / VERL。

**PaperAgent么讲（诚实口径）**

PaperAgent 当前不做微调（API 侧消费外部 LLM）。如果被问"做过大模型微调/训练相关工作吗"，诚实说项目当前是 API 侧消费不做本地微调，但给"如果做"方案（design-only）：

> "PaperAgent 如果要微调，我会选 LoRA——冻结基座 + 注入低秩矩阵（r=16, alpha=32, target_modules=q_proj/v_proj），数据来源是真实用户选题日志 + LLM助生成 + 人工筛选（对标面经里的数据来源四法）。训 conflict detection/feasibility judge 这种领域任务，让模型懂'学术选题'语言。但当前 paperagent 不需要——Minimax API 侧消费 + PromptProtocol + RAG约束已经够用，微调是扩展位不是当前核心卖点。"

讲法引 `docs/interview/Known_Limitations_For_Interview.md`（诚实边界）+ 对应 `Plan/interview_aid/Other_HighFreq_QA.md`（STAR题套话）。

**企业级回答版**

"对标 LLaMAFactory / OpenRLHF / VERL（安克科技岗位明确要求的框架）。SFT vs RLHF：SFT督对齐任务格式，RLHF 类偏好对齐价值观；DPO 是 RLHF化版不需要 reward model。微调 vs RAG：微调让模型懂领域语言（高频术语/风格），RAG供事实和引用，两者结合最优。守'LLM径必须配 heuristic fallback'——微调后的模型也要配 fallback。"

**反问防线**

-问 1："SFT 和 RLHF 有什么区别？" → SFT督学习（input-output 对），RLHF学习（reward model + PPO 优化类偏好）；DPO是简化版无 reward model。
-问 2："简单介绍 LoRA" →结原权重 + 注入低秩矩阵 A/B（rank r），只训 r×d 参数（0.1-1%），显存省 3倍+。常见参数 r/alpha/target_modules。
-问 3："怎么不损失效果做微调？" → LoRA（保基座）+ 数据质量（准确性/多样性/对齐）+ 评估闭环（gold set + 业务指标回归）+底 fallback。
-问 4："RAG 会被微调取代吗？" →不会被取代，而是从"默认必选"降级成"按需启用的一档"。判断：模型强度、文档规模、查询是否需要跨同义词/跨字段。微调让模型懂语言，RAG供事实，结合最优。
-问 5："什么样的项目适合继续做模型重训练，什么样适合只做 RAG 或规则增强？" →需要领域知识内化（高频私域术语、特定风格）→；需要最新事实/可溯源引用/规则可控 → RAG或规则。

**真实问法配精炼答**

1. "SFT 的方案有哪些？" →全参数 / LoRA / Prompt Tuning / Prefix Tuning / QLoRA。
2. "简单介绍 LoRA 技术 / LoRA 中的常见参数" →结基座 + 低秩矩阵 A/B，参数 r/alpha/target_modules。
3. "微调数据集一般从哪里来，怎么判断一批数据值不值得训练？" →四法（开源/标注/LLM助/日志筛选）+量判断（准确性/多样性/对齐/一致性）。
4. "RLHF么做？" → reward model + PPO 优化 + DPO化版。
5. "对微调的理解，SFT 和 RLHF 的区别？" → SFT督学习格式对齐，RLHF化学习偏好对齐。
6. "你过往训练中有没有做过大模型微调/训练相关工作？" → 诚实说 PaperAgent 当前不微调（API 侧消费），给"如果做"的 LoRA 方案。别主动说"不深入"——按面试官反馈调整深度。

### 3.2痛点 #2：多 Agent架构设计薄弱（5表 + Supervisor膨胀到 20-50 最优解）

**原题 + 博主反思**

博主原话："方案设计这一块，之前面试官问方案/架构设计的比较少，这块确实薄弱了一些。最近面试几个大厂，多少还是在讨论方案和架构设计能力了。"小红书场连环追问：6 多 Agent么设计 → 7 用什么多 Agent架构 → 8子 Agent胀到 20-50么优化 → 9.1元宝/豆包面向不同行业子 Agent胀么办 → 9.2型层级分类第一层分错后面全错怎么办 → 9.3票分类准确性但性能开销大怎么办（并行化性能又回到容错/一致性）。

**追问链**

1. "多 Agent时怎么设计？" →先判断是否值得拆（决策树）
2. "子 Agent胀到 20-50么办？" →Hierarchical 降扇出
3. "型层级分类第一层分错后面全错怎么办？" →票 +信度门
4. "票性能开销大怎么办？" →行（同层并行跨层串行）
5. "并行化又回到容错/一致性怎么办？" → Barrier +断 + Saga最终一致

**最优解（5架构表 +合拳）**

5构对比表：

|构 |度 | 适用 |
|---|---|---|
| SAS 单体 | O(T) 顺序，0 通信 |序推理 /具密集 |
| MAS立并行 | O(n) +合/投票 |多视角子任务 |
| 中心化 Supervisor | Supervisor颈 |规划 /发 |
|中心化 Peer |信开销大容错强 |态环境 /索路由 |
| 合 Hierarchical |级 + Peer |用推荐 |

子 Agent胀最优解 = 组合拳不是单点：
1. **Hierarchical 降扇出**：单 Supervisor 50 Agent改多层 Supervisor，每层扇出 b≤5-7。数学：b=5 d=3 = 125量，胜单层线性。
2. **分类用多 Agent票 +信度门**：同层 N轻量分类器 majority/Borda，低置信走人工/回退。
3. **并行 +异步降延迟**：同层并行跨层串行，LangGraph parallel branch，投票别改串行。
4. **速度 vs错一致性取舍**：Barrier 合并快慢差 +显式熔断 closed/half_open/open + negative cache +降级 heuristic + Saga最终一致（不用强一致 2PC，Agent给建议留门不是金融交易）。

量纲：中心化 O(n×k×r) n性涨，Hierarchical度 log_b(n)，投票并发 wall-clock = max(同层) n 串行。

**PaperAgent怎么落**

- **implemented**：当前 SAS + 4 Gate（顺序选题一步错步步错，过早拆多 Agent加汪路由）。7 Agent Role schema +静态路由表 + CostBudget + tally_votes（单多数）。引 `docs/interview/MultiAgent_Expansion_Design.md` + `apps/api/app/services/agent_router.py`（28 个测试）。
- **design-only**：真 Supervisor 编排（当前 mock 数据）+ Hierarchical多层 +并行投票（confidence门）+ LangGraph射（不执行）+显式 CircuitBreaker。引 `Plan/design/MultiAgent_Architecture_Evolution.md` + `docs/interview/AutoResearchClaw_对标与小型化移植.md` §2.7/§3.1/§3.3。

**讲法树**

-（够过）：「PaperAgent是 SAS + 4 Gate，因为顺序选题推理一步错步步错」
- 中（加分）：「Service膨胀时演 Hierarchical，扇出 b=5、度 log_b(n)」
- 深（最优解）：「胀是组合拳：层级降扇出 +同层投票 +并行 +式熔断 + Saga最终一致；不用强一致 Agent给建议人留门」

**3 个触发条件**（什么时候该从 SAS往多 Agent进）：检索源≥5 / 评分模型≥3 /板类型≥5。这 3 条是答"你为什么现在不用多 Agent"的诚实依据（对标 Google 论文 45%值——单 Agent baseline高时加协调收益递减为负）。

### 3.3痛点 #3：量化/推理面试官理解不深（够用关键词 +诚实口径）

**最优解（够用关键词 +一层机制 +诚实口径）**

**必背关键词**：
- 量化方法：INT8/INT4 / AWQ / GPTQ / GGUF Q4_K_M / BF16/FP8
-推理加速：KV-Cache / PagedAttention（vLLM）/ Continuous Batching / Prefix Cache / Speculative Decoding / Chunked Prefill
-推理框架：vLLM / SGLang / llama.cpp / TensorRT-LLM

**一层机制**（被追问展开）：量化本质是用低位表示权重和激活省显存降带宽瓶颈。Q4_K_M是分块混合精度——Attention保 5-bit、FFN 4-bit、输出 6-bit，保关键层精度。KV-Cache存 Transformer 的 K/V免重算历史 token（长序列省平方级开销）。PagedAttention分页管 KV对标 OS 页表解碎片。Prefix Cache用公共 system prompt 的 KV。

**诚实口径（关键，必守）**：PaperAgent 是 API 侧消费（调 Minimax），不做本地量化/推理。被问"你项目怎么量化"时诚实先说，再给"如果做"方案（design-only）：Q4_K_M+ llama.cpp GGUF + prefix cache用同一题目流公共 prompt。收尾："PaperAgent MVP 先把 Agent作流做清楚，量化/推理服务是部署扩展位，不是当前核心卖点。"

**简历包装句（标"推理服务侧方案"不冒充主线）**：
- 「采用 Q4_K_M合精度方案（Attention 5-bit / FFN 4-bit /出 6-bit）实现 CPU端实时推理，显存降约 50%量损失可控」
- 「基于 vLLM PagedAttention + continuous batching署服务，配合 prefix caching用 RAG公共上下文」

**避坑**：别被拽进 GPTQ Hessian 公式细节泥潭；对方深问就诚实说"这块我不是专家，但我清楚量化的应用层取舍和适用场景"，分比硬装高。

## 8. 诚实边界

### 8.1项目不做 /未落地的清单（守 `docs/interview/Known_Limitations_For_Interview.md` 6条边界）

1. **不做本地模型部署 /量化推理**：PaperAgent是 API侧消费（调 Minimax等），不做本地量化 /推理服务。被问"你项目怎么量化"诚实先说，再给"如果做"方案（design-only Q4_K_M + vLLM）。
2. **不做 RAG生产级向量库**：当前是 lightweight（mock BM25 + mock Dense +真 RRF +多因子 Rerank），向量库占位；Qdrant适配层是 design-only。
3. **不做多 Agent Supervisor编排**：当前是 SAS + 4 Gate，`agent_router.py`的是 mock数据。多 Agent Hierarchical是 design-only。
4. **不做 CrossEncoder Rerank /多层验证 L4**：Rerank用多因子 heuristics，CrossEncoder和 L4 LLM判是 design-only；LLM只判 L4相关性，不写 evidence supports。
5. **不做 LangGraph runtime射**：PaperAgent Trace自写支持 replay，LangGraph checkpoint是 design-only对标（引 `MultiAgent_Expansion_Design.md` §4段 D）。
6. **不做 SFT微调**：当前 API侧消费不微调；LoRA / RLHF / DPO是 design-only方案 +对标。

### 8.2讲得不露怯的话术（被挑战时怎么办）

- **"这不就是调 API吗？"** →"PaperAgent核心不是调 API，是把 LLM约束成可审计动作：PromptProtocol + 4 Gate + Evidence软删除留 Trace + readiness hard block + RunEventStore replay。调 API是手段，工程纪律是产出。"
- **"你的 RAG太简单了。"** →"诚实说当前是 lightweight（mock BM25 + Dense +真 RRF +多因子 Rerank）。但接口设计对标 Haystack Pipeline五段可插拔，Qdrant/CrossEncoder/多层验证 L1-L4都是 design-only扩展位，没为短期好看把项目做重。"（引 `RAG_ProfessionalDB_Upgrade.md`）
- **"你这就是单 Agent，多 Agent你会吗？"** →"当前 SAS是有意选择——顺序选题一步错步步错，过早拆多 Agent加汪路由复杂度（对标 Google论文 45%值择）。膨胀时演 Hierarchical +投票 +并行 +断 + Saga最终一致的设计在 `MultiAgent_Architecture_Evolution.md`。"
- **"量化你项目怎么做的？"** →诚实先说"PaperAgent是 API侧消费不做本地量化"，再给"如果做"方案关键词，别硬装做过量化。
- **"你们这个模型用了多少的？"** →诚实说当前走 API不适用量化；给概念答 Q4_K_M含 Attention 5-bit / FFN 4-bit /出 6-bit。
- **"Memory是 LangGraph做的吗？"** →"不是，PaperAgent自写四层记忆 + RunEventStore replay对标 LangGraph checkpoint + time travel。选自写理由是稳定性 /可测 /可审计，LangGraph runtime是 design-only射。"
- **"你用过 LangChain / LlamaIndex / LangGraph吗？"** →诚实说读过源码 +对标思路，当前自写主线，留接口不锁死。
