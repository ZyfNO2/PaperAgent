## 1. ResearchAgent

**URL**: https://github.com/JinheonBaek/ResearchAgent
**类型**: 文献驱动的迭代式科研创意生成 (arXiv 2404.07738, NAACL 2025)
**技术栈**: Python 3.9+ · OpenAI GPT-4o · Semantic Scholar Graph API · JSONL

**核心架构**: 单根 paper 出发 → 拉 references + 实体知识库 → Problem / Method / Experiment 三段串行 pipeline，每段跑 N 轮"生成 → 多准则评审 → 选最优"迭代；评审用 `ThreadPoolExecutor` 并发 5 个 LLM call，最后用 `max(history, key=get_avg_score)` 挑最佳。

**关键设计点**:
- **三阶段管道（Problem → Method → Experiment）**：每段独立迭代，下游消费上游最佳输出
- **5 维并行评审**（Clarity/Relevance/Originality/Feasibility/Significance），用 `ThreadPoolExecutor(max_workers=5)` 一次性发
- **正则解析 LLM 输出**：`Review:\s*(.*?)\nFeedback:\s*(.*?)\nRating(?:\s*\(1-5\))?:\s*([1-5])`，强制三段式结构化文本
- **history dict 累积**：每轮 `{problem, rationale, feedbacks}` 全部保留，便于 ablation

**有意思的代码片段**:
```python
# problem_validator.py: 并发评审 + 结构化解析
with ThreadPoolExecutor(max_workers=len(self.build_functions)) as executor:
    futures = {
        metric: executor.submit(self._chat, user_prompt=function(context))
        for metric, function in self.build_functions.items()
    }
    feedbacks = {m: self.parse_output(f.result()) for m, f in futures.items()}

# parse_output: 用正则锁死 LLM 输出 schema
match = re.search(
    r"Review:\s*(.*?)\nFeedback:\s*(.*?)\nRating(?:\s*\(1-5\))?:\s*([1-5])",
    text, re.DOTALL | re.IGNORECASE
)
```

**对 TopicPilot-CN 启示**:
- Phase 07 的 committee review 可以照搬 ThreadPoolExecutor + 5 维并发模式
- "强制 LLM 输出 schema + 正则解析" 比 JSON-mode 更稳，可作 heuristic fallback 的备选
- 三段级联正好映射 Phase 02→03→04 的数据流，每个 Phase 选 best-of-N 写入 context

## 2. IRIS (Interactive Research Ideation System)

**URL**: https://github.com/Anikethh/IRIS-Interactive-Research-Ideation-System
**类型**: MCTS 驱动的可交互式假设生成 (ACL 2025 Demo)
**技术栈**: Python · Flask · SocketIO · LiteLLM · Semantic Scholar · Google Gemini · HuggingFace cross-encoder reranker

**核心架构**: Flask + SocketIO 提供实时 UI，后端把"idea 探索"建模成 Monte Carlo Tree Search——每个节点是 idea 状态，边是 generate / judge / review / retrieve 四类动作，通过 UCT 选择、扩展、奖励反传，最终给 best_child；前端用 WebSocket 流式推送并把整棵树可视化。

**关键设计点**:
- **MCTS over idea space**：UCT 选节点 → expand → review_agent 打 reward → γ 反向回填 Q/N
- **fine-grained feedback**：用户对单个 aspect（novelty/feasibility 等）单独打意见，注入 MCTS 当 reward
- **ScholarQA + HF reranker 做 RAG**：Semantic Scholar 粗检 + cross-encoder 重排
- **PDF/文本上传 → PyMuPDF 解析**：用户本地 paper 当额外 knowledge source

**对 TopicPilot-CN 启示**:
- Phase 06/07 的"委员会多轮对话"可借鉴 MCTS：每个委员观点当 child node，多轮 refinement 当 backpropagation
- IRIS 的 `/api/tree` + `/api/node` 双端点正好是 apps/web 右侧栏要做的"探索路径可视化"模板
- "fine-grained feedback" 可落到 Phase 07——让用户对 EvidenceLedger 的某条单独点赞/否决，触发该路径的局部重生成

## 3. Idea2Proposal

**URL**: https://github.com/NuoJohnChen/Idea2Proposal
**类型**: 多 Agent 学术讨论 + 8 维评审 (arXiv 2508.04575, Apache 2.0)
**技术栈**: Python 3.9+ · DeepSeek V3 / GPT-4 / O1-mini · Semantic Scholar · YAML 配置驱动

**核心架构**: 完全 YAML 配置驱动——`prompts / environment / agents / tools / ai_researcher_config` 五大块定义一次学术讨论，5 种协作范式（Horizontal / Vertical / Interdisciplinary / Leader_Led / Multi）+ 2 种单人反思模式；评审走 3 ensemble reviews → meta-review → self-reflection → weighted average。

**关键设计点**:
- **8 维评审**（1-10 分 + Overall Quality）：Novelty / Workability / Relevance / Specificity / Integration_Depth / Strategic_Vision / Methodological_Rigor / Argumentative_Cohesion
- **Ensemble + Self-Reflection 双保险**：并发 3 份独立评审 → meta-review → 3 轮 self-reflection → 加权平均
- **YAML 即对话剧本**：每种 collaboration pattern 是独立文件夹，启动时读 yaml
- **YAML `agents` 段定义 persona**：每个 agent 一份 system prompt 模拟 senior / junior / cross-domain

**有意思的代码片段**:
```python
review = perform_structured_review(
    proposal_text, model="deepseek-v3", client=openai_client,
    temperature=0.1, num_reviews_ensemble=3, num_reflections=3,
)
# 内部: 1) 并发 3 份 8 维评分 2) meta-review 合成 3) 3 轮反思 4) 加权平均 → final + Decision
```

**对 TopicPilot-CN 启示**:
- TopicPilot 7 维可扩到 8 维（补 Integration_Depth + Strategic_Vision）
- YAML-driven + 5 协作模式 = 后续加评审模式无需改 Python 代码
- Ensemble (3) + Reflection (3) 可作 Phase 04 LLM 稳定性兜底

## 4. research-companion

**URL**: https://github.com/andrehuang/research-companion
**类型**: 战略评估（adversarial idea critic）
**技术栈**: Claude Code / Codex 插件 · Markdown agent prompt · YAML manifest · MEMORY.md 持久化

**核心架构**: 三 agent + 一 skill 编排。三 agent = `idea-critic`（7 维度评分，给 PURSUE / REFINE / KILL 判决）、`research-strategist`（项目级 continue / pivot / kill + 影响预测）、`brainstormer`（跨领域联想 + 假设挑战 + 怀疑读者测试）。`research-companion` skill 把 3 个 agent 串成 6 阶段管线：Seed → Diverge → Evaluate → Deepen → Frame → Decide。

**关键设计点**:
- **7 维度评分表**：每维 enum（Weeks/Months/Years；Low/Med/High；Too Early/Well-Timed/Too Late；Crowded/Moderate/Open；Clear/Fuzzy/Missing；Compelling/Workable/Weak），强结构化、机器可解析
- **结论先行测试（Carlini）**：在 Phase 5 强制先写 abstract + 2-3 句 conclusion，否则算 idea 不成熟
- **持久化到 `research-evaluations/YYYY-MM-DD-<slug>.md` + MEMORY.md 索引**，下次同主题主动查 prior eval
- **每条 idea 配 "One Question to Resolve Next"**：必须是可测试的
- **agent frontmatter 用 `model: opus` + 显式声明 `tools: Read, Glob, Grep, WebSearch, WebFetch`**

**对 TopicPilot-CN 启示**:
- Phase 03 检索计划生成后，可借鉴 7 维度评分再做 idea-critic 式 stress test
- `MEMORY.md` + `research-evaluations/` 持久化模式可原样搬过来做"过去跑过这个 topic 的历史"查询，避免重复评估

## 5. idea-evaluation-pipeline

**URL**: https://github.com/alejandroll10/idea-evaluation-pipeline
**类型**: 评估-复核-转向（evaluate → review → pivot loop）
**技术栈**: 纯 prompt 文件 · Opus / Sonnet 多 agent · WebSearch

**核心架构**: 8 步循环管道，**每个有"评分"的步骤后面都跟一个"评审"步骤**。流：`1. EVAL → 2. REVIEW EVAL → (≥7?) → 5. LIT REVIEW → 6. VERIFY LIT REVIEW（带 URL 校验）→ 7. FINAL VERDICT → 8. REVIEW VERDICT → (≥7?) DONE`，否则回 3. PIVOT → 4. EVAL PIVOT 再循环。

**关键设计点**:
- **3 闭环嵌套**：评测闭环 (1-2) / pivot 闭环 (3-4) / 终判闭环 (7-8)，每个都要求 ≥7/10 才放行
- **强制 URL 引用**：step 6 verify_lit_review 要求 agent 用 Edit 权限**直接改文件**，给每篇 paper 加 Google Scholar/SSRN URL，未验证的标 `[UNVERIFIED]` 或删除——抗幻觉硬约束
- **迭代版本号**：pivot 失败带 "full history" 重做，文件名 `pivot_idea1_v2.txt` 显式标注轮次
- **LIT REVIEW 专门搜"威胁论文"**（审稿人会引用的那种）
- **target score 7/10 写死在 prompt 头部**

**有意思的代码片段**:
```
# pivot_prompt.txt 的 6 条硬约束
1. Keep the core topic — do not abandon what the student cares about.
2. Add a credible identification strategy — name the specific instrument or natural experiment.
3. Sharpen the theoretical prediction — state sign, mechanism, alternatives.
4. Propose specific regressions and data — write the regression equation, name every dataset.
5. Articulate "what's new" in one paragraph.
6. If 2nd+ pivot: explain what went wrong with previous pivot.
```

**对 TopicPilot-CN 启示**:
- Phase 04 EvidenceLedger 可学"生成 + 评审"两步走——LLM 产 evidence 后再起 critic 评审，强制每条 evidence 配 arXiv URL
- `pivot_idea[N]_v[N]` 模式可原样搬过来：Phase 03 SearchQueryPlan 评分不过应支持 `_v2 / _v3` 迭代版本号
- "威胁检索"可作为 Phase 03 新检索层加入

## 6. DatasetResearch

**URL**: https://github.com/GAIR-NLP/DatasetResearch
**类型**: 数据集发现 (demand-driven dataset discovery benchmark)
**技术栈**: Python 3.10+ · uv · LLaMA-Factory SFT · OpenAI / Azure OpenAI 兼容 · HuggingFace Datasets · YAML 配置

**核心架构**: 6 子命令 CLI（`search` / `synthesis` / `register` / `run` / `eval` / `metaeval`），核心算法 `scripts/method/search_data.py` 的 4 阶段流水线：search_agent → check_exist_gated → generate_search_train_set → generate_search_set_metadata。

**关键设计点**:
- **数据集 ID 规范化**：`search_{selected_dataset}_{model}`，所有产物文件名即元数据
- **YAML 配置优先于 env var**：所有 API key、模型名、pipeline 参数都走 YAML
- **Enhanced CLI + 降级**：`try: from ... import cli; ENHANCED_CLI = True; except ImportError: ENHANCED_CLI = False`，富文本失败时降级到 `print()`
- **dry-run 一等公民**：每个 command 都带 `--dry-run` flag
- **Full pipeline 强制 confirm**：`run --step full` 调 `cli.confirm_action()` 弹确认
- **gated 检查**：`check_exist_gated` 用 `huggingface_hub.dataset_info` + `list_repo_files` 校验

**有意思的代码片段**:
```python
# Enhanced CLI pattern — graceful degradation
try:
    from datasetresearch_cli.utils.cli_utils import cli
    ENHANCED_CLI = True
except ImportError:
    ENHANCED_CLI = False

def handle_search(cli_manager, args):
    if hasattr(args, 'dry_run') and args.dry_run:
        if ENHANCED_CLI:
            cli.print_info("Dry run mode - showing command that would be executed:")
            cli.console.print(f"  {' '.join(cmd)}", style="cyan")
            return 0
        # ... 实际执行
```

**对 TopicPilot-CN 启示**:
- **YAML 配置优先**：pipeline 参数（检索层数、并发数、每层 top_k）从代码移到 `configs/pipeline_settings.yaml`
- **Enhanced CLI + 降级**：Phase 01-04 端点如果 fail 不应 500，用 `try import rich except ImportError` 降级
- **dry-run 一等公民**：每个 API 端点提供 `?dry_run=1`，build 出 plan 后 return 不执行
- **gated 检查**：调 arXiv API 前先 HEAD 一下，看 query 是否返回空

## 7. ResearchMCP

**URL**: https://github.com/DaniManas/ResearchMCP
**类型**: MCP 工具（学术论文检索）
**技术栈**: Python 3.12+ · FastMCP · OpenAlex API (250M+ 论文, 无需 API key) · httpx · uv

**核心架构**: 单进程 MCP server，分两层 — `server.py` 仅声明 `@mcp.tool()` 装饰的工具（6 个），所有网络与数据解析下沉到 `tools/paper_fetcher.py` 的 `PaperFetcher` 类。工具函数体不写逻辑，只负责调用 fetcher、拼 Markdown 文本、把"指令"塞进返回值让 Claude 自行推理。

**关键设计点**:
- **MCP tool 返回纯 Markdown 字符串**，把"分析指令"作为 suffix 拼到数据后面
- **OpenAlex 而非 arXiv**，覆盖 250M+
- **6 个工具覆盖论文生命周期**: 搜索 / 拉摘要 / 提 claims / 对比 2-5 篇 / 引用图 / 找研究 gap
- **`find_research_gaps` 用 `sort_by="cited_by_count"`** 优先取高被引论文
- **错误用 dict `{"error": ...}` 统一传递**

**有意思的代码片段**:
```python
@mcp.tool()
def extract_claims(paper_id: str) -> str:
    paper = fetcher.fetch_paper_by_id(paper_id)
    if "error" in paper: return paper["error"]
    abstract_text = fetcher.get_paper_abstract(paper)
    result = f"**Paper:** {paper['title']}\n"
    result += f"**Abstract:**\n{abstract_text}\n\n"
    result += f"**Instructions for claim extraction:\n"
    result += f"Please analyze the abstract above and extract:\n"
    result += f"1. Main research question or hypothesis\n"
    result += f"2. Key methodology or approach\n"
    result += f"3. Primary findings or results\n"
    result += f"4. Main conclusions or implications\n"
    return result
```

**对 TopicPilot-CN 启示**:
- Phase 03 arXiv 检索可借鉴 `search_papers` 的分页与年份过滤参数化设计
- 我们的"LLM 必须配 heuristic fallback"原则正好对应"工具自身不依赖 LLM"思路
- `compare_papers` 的"对比 2-5 篇"模式可搬到 Phase 04 多源对比

## 8. RAG_Research_Paper_Gap_Finder

**URL**: https://github.com/bistadinank/RAG_Research_Paper_Gap_Finder
**类型**: 本地 RAG + 研究 gap 识别
**技术栈**: Python 3.9+ · Ollama (Llama 3.2) · ChromaDB · pypdf · sentence-transformers (`all-MiniLM-L6-v2`, 384-dim) · 余弦相似度

**核心架构**: 7 步逐步构建的"教科书式"流水线 — `01_load_pdf` → `02_chunk`（overlap 切分）→ `03_embed` → `04_store` → `05_rag` → `06_multi_store`（section-aware 元数据）→ `07_gap_finder`（双路检索 + LLM）。

**关键设计点**:
- **Section-aware chunking**: 元数据带 `{"paper": ..., "section": "limitations|future work|conclusion|..."}`
- **双路检索 = 高信号窄检索 + 宽泛语义检索**。窄路只查 6 个"作者承认未解"的 section，各取 top-3
- **去重用前 100 字符作 key**
- **Prompt 设计** — 不是"文献说了什么 X"，而是"文献 **没** 解决什么 X"
- **完全本地** — Ollama + ChromaDB，无 API 成本

**有意思的代码片段**:
```python
GAP_SECTIONS = ["limitations", "future work", "conclusion", "conclusions",
                 "discussion", "results and discussion"]
gap_chunks = []
for section in GAP_SECTIONS:
    try:
        results = collection.query(
            query_embeddings=topic_embedding,
            n_results=3,
            where={"section": section},
            include=["documents", "metadatas"]
        )
        for doc, meta in zip(results["documents"][0], results["metadatas"][0]):
            gap_chunks.append({"text": doc, "paper": meta["paper"], "section": meta["section"]})
    except Exception:
        pass
```

**对 TopicPilot-CN 启示**:
- Phase 04 EvidenceLedger 可借鉴 section-aware 元数据：把 arXiv 全文按 Abstract/Method/Results/Limitations 分块
- "双路检索" 适合"宽泛背景 + 精确证据"混合查询
- "打印所有 retrieved chunk 给用户肉眼验证" 是好 UX，可放进 apps/web 右侧 trace 面板

## 9. ARIS (auto-claude-code-research-in-sleep)

**URL**: https://github.com/wanshuiyin/auto-claude-code-research-in-sleep
**类型**: 多 Skill 调度 + 跨模型对抗评审
**技术栈**: Markdown (SKILL.md) · Python 工具脚本 · Rust CLI · Claude Code · Codex / Oracle / Gemini MCP · GitHub Actions

**核心架构**: **纯 Markdown skills 集合**（77+ 个 `SKILL.md`），无数据库、无 Docker。Research Wiki 是核心 — 持久化、按项目划分的知识库，4 类节点（Paper/Idea/Experiment/Claim）+ 8 类有向边（`extends` / `contradicts` / `addresses_gap` / `supports` / `invalidates`）。

**关键设计点**:
- **Research Wiki 灵感来自 Karpathy LLM Wiki**：知识累积、跨会话持久
- **`query_pack.md` 硬预算 8000 字符 (~2000 tokens)**：分 6 段（Top 5 gaps / Paper clusters / Failed ideas / Top papers / Active chains / Open unknowns）
- **失败的想法 = 最有价值的记忆**：永远不被剪枝
- **跨模型对抗评审**：Claude Code 主执行 + Codex / Oracle MCP 当 reviewer 挑刺
- **`verify_papers.py` Pre-Search Verification Protocol**：反幻觉
- **Canonical helper 链**：通过 `git rev-parse` → `.aris/installed-skills.txt` → `tools/research_wiki.py` 四级 fallback

**有意思的代码片段**:
```bash
# Helper 解析链（shell, 无 Python 依赖）
cd "$(git rev-parse --show-toplevel 2>/dev/null || pwd)" || exit 1
ARIS_REPO="${ARIS_REPO:-$(awk -F'\t' '$1=="repo_root"{print $2; exit}' .aris/installed-skills.txt 2>/dev/null)}"
WIKI_SCRIPT=".aris/tools/research_wiki.py"
[ -f "$WIKI_SCRIPT" ] || WIKI_SCRIPT="tools/research_wiki.py"
[ -f "$WIKI_SCRIPT" ] || { [ -n "${ARIS_REPO:-}" ] && WIKI_SCRIPT="$ARIS_REPO/tools/research_wiki.py"; }
[ -f "$WIKI_SCRIPT" ] || { echo "ERROR: research_wiki.py not found" >&2; exit 1; }
```

**对 TopicPilot-CN 启示**:
- **`query_pack.md` 8000-char 硬预算模式**可移植到 Phase 04-08 的 LLM 上下文
- **"failed ideas 永不剪枝"** 对应 Phase 06 的 workpackage pivot 决策记录
- **跨模型 reviewer 思路** 适合 Phase 07 委员会对话
- **Canonical helper 链** 适合 Phase 04 的 evidence 持久化

## 10. AutoResearchClaw

**URL**: https://github.com/aiming-lab/AutoResearchClaw
**类型**: 端到端自动科研 (从主题输入到 LaTeX 论文)
**技术栈**: Python 3.11+ · Pydantic · OpenAI 兼容 · ACP 协议 (Claude Code / Codex) · Docker 沙箱 · LiteLLM · arXiv / Semantic Scholar / OpenAlex

**核心架构**: 23 阶段流水线（A 范围 → B 文献 → C 综合 → D 设计 → E 执行 → F 决策 → G 写作 → H 定稿），每个阶段有 Pydantic 契约、`StageStatus` 状态机、`TransitionEvent` 事件驱动迁移，带 3 个 HITL 门控 (Stage 5/9/20) + Stage 15 PIVOT/REFINE 决策节点。

**关键设计点**:
- **PIVOT 决策回滚映射**: `DECISION_ROLLBACK = {"pivot": HYPOTHESIS_GEN, "refine": ITERATIVE_REFINE}` + `MAX_DECISION_PIVOTS=2`
- **门控阶段可配置**: `GATE_STAGES = {5, 9, 20}`
- **VerifiedRegistry 反造假**: 论文中数字强制与实验数据对齐，SHA256 版本快照
- **SmartPause 动态置信度暂停**: HITL 自动判断"何时打断人"
- **MetaClaw 跨运行学习**: 失败 → 经验 → 技能

**有意思的代码片段**:
```python
# 状态机迁移: REJECT 触发 PIVOT, 回退到目标阶段
if event is TransitionEvent.REJECT and status is StageStatus.BLOCKED_APPROVAL:
    return TransitionOutcome(
        stage=target_rollback, status=StageStatus.PENDING,
        rollback_stage=target_rollback,
        checkpoint_required=True, decision="pivot",
    )

def gate_required(stage, hitl_required_stages=None, *, profile=None):
    is_gate = stage in GATE_STAGES
    is_hep_ph_codegen_gate = (stage is Stage.CODE_GENERATION and profile == "hep_ph")
    if is_hep_ph_codegen_gate:
        return True  # 强约束, 即使 hitl_required_stages=None 也不放过
```

**对 TopicPilot-CN 启示**:
- Phase 03(检索)失败可回退到 Phase 02(拆解)而非全流程重启
- `MAX_DECISION_PIVOTS=2` 这种"硬上限"比单纯 try/retry 更适合生产
- 门控判断 (gate_required) 可作 Phase 01-08 评级硬约束的统一接口

## 11. ResearchRubrics

**URL**: https://github.com/scaleapi/researchrubrics
**类型**: 深度研究 Agent 的 Rubric 评估管线 (ICLR 2026 论文代码)
**技术栈**: Python 3.8+ · LiteLLM (Gemini 2.5 Pro) · asyncio · pandas · JSONL · HuggingFace

**核心架构**: 4 段流水线（`processed_data.jsonl` 拉取 → `evaluate_rubrics` LLM 评分 → `calculate_metrics` 加权聚合 → 输出 Compliance Score）。长文档按段落切成 ≤100K token 块，逐块调用 LLM 收集 evidence，最后 synthesis prompt 汇总。

**关键设计点**:
- **加权合规分公式**: `Compliance = Σ(score × weight) / Σ(weight>0)`，负权重自动从分母排除
- **二元判定 + 置信度**: verdict 强制 YES/NO/NA，score ∈ {0.0, 1.0}，额外存 `confidence` / `tokens_used` / `cost`
- **asyncio.Semaphore(20) + 指数退避**: `await asyncio.sleep(2 ** attempt)` (1s/2s/4s)
- **`response_format={"type": "json_object"}` 强约束**
- **chunk + synthesis 两阶段 prompt**

**有意思的代码片段**:
```python
for attempt in range(3):
    try:
        response = await litellm.acompletion(
            model="litellm_proxy/gemini/gemini-2.5-pro-preview-06-05",
            messages=[{"role": "system", "content": SYSTEM_PROMPT},
                      {"role": "user", "content": user_prompt}],
            max_tokens=50000,
            response_format={"type": "json_object"},
        )
        result_data = json.loads(response.choices[0].message.content)
        return EvaluationResult(
            score=float(result_data["score"]),
            confidence=float(result_data["confidence"]),
            cost=self._calculate_cost(response.usage),
            success=True,
        )
    except Exception as e:
        if attempt == 2: raise
        await asyncio.sleep(2 ** attempt)  # 1s → 2s → 4s
```

**对 TopicPilot-CN 启示**:
- `response_format=json_object` + 指数退避 + `Semaphore(20)` 是 LiteLLM 调用的"三件套"模板
- `confidence` 字段是低成本质量信号，可加到 EvidenceLedger 自动过滤低置信引用
- chunk → synthesis 两阶段评估适合 Phase 08 最终材料渲染（长 proposal 切块评估）
