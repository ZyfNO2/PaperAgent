# PaperAgent 后续 Session 21-26 简易 SOP 计划：边界约束与主线确定

> 日期：2026-06-20  
> 定位：这是 Session21 之后的短期执行护栏，不替代各 Session 的详细 SOP。  
> 核心目的：防止项目在“酒馆式渲染、Agent 化、证据工作台、开题报告生成”之间发散，先把选题 MVP 主线跑顺。

---

## 1. 当前主线

短期主线只保留一句话：

```text
用户输入一个题目或上传少量材料，系统逐步拆解选题，暂停让用户审查关键词，再检索论文/数据集/工程，最后给出可行性判断和开题报告初稿建议。
```

第一阶段体验必须是：

```text
输入题目
-> 流式题目理解
-> 关键词拆解卡片
-> 暂停等待用户审查
-> 用户修改 / 确认
-> 继续进入检索计划
```

不要一开始追求完整 Agent。先让用户感觉：

```text
我知道系统现在分析到了哪一步；
我能在关键节点改方向；
我能看到每个结论从哪里来；
系统不会一口气吐出一页难定位的报告。
```

---

## 2. 总边界

### 2.1 不能动的底线

```text
1. 不破坏 S17 Demo baseline；
2. 不绕过 EvidenceRef / Verification / supports 规则；
3. 不让 LLM 输出任意 HTML / JS 后直接执行；
4. 不把未经验证的网页、论文、数据集直接当作强证据；
5. 不把“可交互渲染”做成不可审计的黑箱；
6. 不在没有测试的情况下连续推进多个高风险 Session；
7. 不把短期 MVP 做成长期难维护的临时拼接。
```

### 2.2 允许先简化的地方

```text
1. 检索结果可以先用 mock / demo fixture；
2. LLM 流式可以先用 mock stream；
3. 组件库可以先只做 3-5 个核心卡片；
4. 证据工作台可以先做只读对照，不急着拖拽；
5. 开题报告可以先输出结构化草稿，不急着做 DOCX；
6. Agent 助手可以先做“生成卡片草案”，不直接改证据链。
```

### 2.3 必须持续保留的审计点

```text
1. 用户改了哪些关键词；
2. 系统用哪些关键词检索；
3. 哪些候选被接受 / 淘汰；
4. 哪些证据支持哪个结论；
5. 哪些内容只是建议，不是已验证事实；
6. 每次 Gate 暂停与继续的时间和原因。
```

## 3. Session 21：Step Deck + 流式关键词 Gate

详细 SOP：

```text
Plan/PaperAgent_Session21_分步滑动流式交互与酒馆式渲染SOP.md
```

目标：

```text
把单页高密度报告改成分步滑动工作流；
用流式事件驱动题目理解和关键词拆解；
在 keyword_review 后强制暂停。
```

最小完成标准：

```text
1. 页面是 Step Deck，不再是一页长报告；
2. 题目理解能流式出现；
3. 关键词卡能流式渲染；
4. 用户能增删改关键词；
5. 确认后进入 query_plan；
6. 非法 paperagent-card 不执行脚本；
7. Playwright 覆盖暂停与继续。
```

停止边界：

```text
做到 query_plan 占位页即可停；
不要在本 Session 内继续做完整检索、开题报告、双栏工作台。
```

## 4. Session 22：Renderer Component Registry 最小卡片库

目标：

```text
把 Session21 中临时可用的卡片协议沉淀为稳定白名单组件库。
```

核心卡片：

```text
TopicUnderstandingCard
KeywordReviewCard
SearchQueryPlanCard
RetrievalCandidateCard
EvidenceRefCard
ReportQualityCard
```

要做：

```text
1. 定义 component registry；
2. 每个组件有 schema；
3. 每个 action 有固定 action_id；
4. 非法组件统一降级；
5. 前端和后端共享或镜像校验规则；
6. 补 parser / renderer 单测。
```

不做：

```text
不做复杂插件市场；
不允许用户自定义任意 JS 组件；
不接第三方 iframe demo；
不做完整主题皮肤系统。
```

验收：

```text
合法组件稳定渲染；
非法组件安全降级；
action 不再散落在页面逻辑里；
Playwright 覆盖至少 3 类卡片互动。
```

## 5. Session 23：流式 Prompt 协议与工具调用边界

目标：

```text
让 LLM 输出从“随意 Markdown”变成“自然语言 + 受控 paperagent-card”的混合协议。
```

要做：

```text
1. 定义 LLM 输出规范；
2. 明确何时输出 token_delta，何时输出 card_delta；
3. 给每个 Step 写 prompt skeleton；
4. 增加 render block 失败回退；
5. 定义工具调用前必须经过哪些 Gate；
6. 给 prompt 加安全条款：不得输出 script / iframe / onClick。
```

不做：

```text
不接入多个真实大模型；
不做自动联网深度研究；
不让 LLM 直接写 Evidence Ledger；
不让 LLM 跳过用户确认继续检索。
```

验收：

```text
同一个题目多次运行时，输出结构稳定；
模型输出坏 JSON 时页面不崩；
Prompt 能稳定产出关键词卡、检索计划卡、候选证据卡。
```

## 6. Session 24：检索计划与候选资源卡

目标：

```text
从确认后的关键词出发，生成论文、数据集、工程三个方向的检索计划和候选卡。
```

要做：

```text
1. 把关键词映射为检索 query；
2. 候选资源分为 paper / dataset / repo；
3. 每个候选都有来源 URL、摘要、匹配关键词、初步可用性；
4. 候选只进入 Candidate，不直接成为 Evidence；
5. 用户可标记：有用 / 暂存 / 淘汰 / 需要复核。
```

不做：

```text
不承诺所有 URL 都真实可访问，除非经过 URLVerified；
不把候选资源自动写进开题报告；
不做大规模爬虫；
不做 PDF 全文解析。
```

验收：

```text
确认关键词后能看到三类候选卡；
每张卡能解释为什么被推荐；
用户标记行为进入 Trace；
候选和正式证据有清晰区分。
```

## 7. Session 25：双栏证据工作台 MVP

目标：

```text
实现用户灵感中的“左边是用户想用的资料，右边是系统搜到的资料”的最小工作台。
```

界面：

```text
左栏：用户指定 / 收藏 / 准备用的论文、数据集、工程；
右栏：系统检索到的候选；
中间或底部：当前选题可行性判断。
```

要做：

```text
1. 支持把候选加入左栏；
2. 支持从左栏移除；
3. 支持标记核心资料；
4. 支持查看 EvidenceRef / URLVerified / Trace；
5. 支持按论文、数据集、工程筛选。
```

不做：

```text
不做复杂拖拽也可以，按钮加入即可；
不做多人协作；
不做自动写论文正文；
不让未验证资料成为强支撑。
```

验收：

```text
用户能清楚区分“我选中的资料”和“系统候选资料”；
加入 / 移除 / 标核心都有记录；
工作台不会破坏原有证据规则。
```

---

## 8. Session 26：开题报告推荐与低门槛复核

目标：

```text
基于关键词、候选资源、用户选中的核心资料，生成开题报告初稿建议，并做一次低门槛委员会复核。
```

要做：

```text
1. 输出题目理解；
2. 输出研究背景；
3. 输出研究内容；
4. 输出技术路线；
5. 输出可行性分析；
6. 输出风险与替代方向；
7. 输出低门槛审稿意见：是否像一个能开题的方向。
```

不做：

```text
不写完整毕业论文；
不承诺学术创新已经成立；
不自动虚构参考文献；
不把低门槛复核包装成真实专家意见。
```

验收：

```text
报告每个关键判断都能追溯到关键词、候选资源或用户选择；
低门槛复核能指出明显问题；
报告可作为后续正式开题报告的素材，而不是最终稿。
```

## 9. 后续待定方向

这些方向先保留，不进入 21-26 的主线：

```text
1. Agent 助手根据网页链接、图片、文字描述自动生成卡片；
2. 自定义 Skill 下载和运行；
3. 多模型对照评审；
4. 文献全文 RAG；
5. DOCX / PPT 自动导出；
6. 多项目管理；
7. 长期记忆和用户偏好；
8. 更接近 SillyTavern 的脚本生态。
```

进入条件：

```text
只有当 Session21-26 的主线稳定、测试通过、验收报告明确无阻塞后，再逐个纳入。
```

## 10. 每个 Session 开工前检查

每次开新 Session 前，先回答：

```text
1. 这一轮是否服务“选题 MVP 主线”？
2. 是否会破坏 S17 baseline？
3. 是否需要新增 Playwright 测试？
4. 是否需要新增后端测试？
5. 是否会改变 EvidenceRef / Verification / Trace 语义？
6. 是否存在任意 JS / HTML 注入风险？
7. 是否需要更新 SOP 或验收报告模板？
```

如果答案不清楚，先不写代码，先补 SOP。

---

## 11. 每个 Session 完工报告固定结构

```text
1. 本轮目标；
2. 实际完成；
3. 未完成；
4. 主线是否偏移；
5. 边界是否被突破；
6. 测试结果；
7. Playwright 结果；
8. S17 baseline 是否通过；
9. 风险和回滚建议；
10. 下一轮是否可以继续。
```

## 12. 当前推荐执行顺序

```text
Session 21：Step Deck + 流式关键词 Gate
```

低风险并行建议：

```text

Session 22 可以紧跟 21；
Session 23 最好在 22 后做；
Session 24 必须等 keyword gate 和 prompt 协议稳定；
Session 25 必须等候选资源卡稳定；
Session 26 必须等双栏工作台能明确用户选择。
```

---

## 13. 最终提醒

```text
主线不是“做一个很炫的 Agent UI”，而是“让学生能一步步把一个题目变成可判断、可调整、可追溯的开题方向”。

交互渲染是手段；
关键词 Gate 是第一道闸；
证据可追溯是底线；
开题可行性判断才是短期结果。
```
