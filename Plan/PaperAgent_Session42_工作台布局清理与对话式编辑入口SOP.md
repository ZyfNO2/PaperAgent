# PaperAgent Session 42 SOP：工作台布局清理、Trace 折叠与对话式编辑入口

## 0. 本次定位

Session 41 已经把“一次性全出”改成了分步工作台，但人工验收截图暴露出新的 UI 问题：

1. 左右两栏的职责需要互换：**左侧放 LLM 思维 / 对话，右侧放证据 Trace**。
2. 中间下方仍保留了 1-5 步的重复大卡片，和上方分页工作区重复，占用大量空间。
3. `▶ 开始 Step 1: 题目理解` 这类 Trace 事件需要支持折叠，否则 Trace 很快变成长列表。
4. LLM 思维 / 对话区缺少一个真正的对话入口，用户无法通过自然语言引导 LLM 修改证据、步骤、关键点和工作台元素。

本 Session 的目标不是扩展新 Agent 能力，而是把工作台整理成可长期使用的人机协作界面。

## 1. 本次主线

将页面从“分步展示面板”升级为“可对话编辑的研究工作台”：

```text
左侧：LLM 思维 / 对话 / 工作台指令入口
中间：当前 Step 工作区 + 横向分页
右侧：证据 Trace / 证据状态 / 可折叠执行记录
底部：仅保留开题报告导出，不再显示 Step 1-5 重复大卡片
```

关键要求：

- 左右互换后，翻页仍然只影响中间 Step 工作区。
- LLM 对话入口常驻左侧底部。
- 用户可以通过对话对工作台元素进行增删改查，但所有写操作必须先预览，再确认。
- 右侧 Trace 按 Step 分组折叠。
- 删除或隐藏中间下方多余的 Step 1-5 大卡片，仅保留 Step 6 导出区域。

## 3. 多余元素清理

### 3.1 需要清理的位置

截图中，中间工作区下方还有这些重复大卡：

- `1 题目理解`
- `2 关键词拆解`
- `3 证据检索（论文 / 数据集 / 工程）`
- `4 可行性判断`
- `5 开题建议 & 低门槛审核`
- `6 开题报告导出（Markdown）`

其中 1-5 已经被上方 Step 工作区和横向分页替代，继续保留会造成：

- 页面高度过长。
- 用户误以为需要继续往下操作。
- 信息结构重复。
- 工作台主焦点被打散。

### 3.2 清理规则

整改要求：

- 删除或默认隐藏 Step 1-5 大卡片。
- 保留 Step 6 导出区域。
- Step 6 导出区域改成紧凑卡片或折叠区。
- Step 6 只有在 Step 5 完成后才高亮可用。
- 如果 Step 1-5 仍需作为流程总览，必须收进一个默认折叠的 `流程总览` 抽屉，不得直接铺在页面下方。

推荐方案：

```text
当前 Step 工作区
└─ 紧凑导出区
   ├─ 开题报告导出（Markdown）
   ├─ 导出前状态
   ├─ 模板选择
   └─ 生成报告
```

验收要求：

- 首屏和主工作区下方不再看到 Step 1-5 重复大卡。
- Step 6 导出仍可见或可展开。
- 导出按钮不回退。
- 页面高度明显降低。

## 4. 右侧证据 Trace 折叠

### 4.1 Trace 分组结构

右侧 Trace 不再是单条事件无限列表，而是按 Step 分组：

```text
证据 Trace
├─ Step 1：题目理解              已完成  6 条事件
├─ Step 2：关键词拆解            已完成  5 条事件
├─ Step 3：检索计划与候选证据    已完成  8 条事件
├─ Step 4：可行性判断            已完成  4 条事件
└─ Step 5：开题建议              已完成  4 条事件
```

每组展开后显示该 Step 的事件。

### 4.2 折叠规则

必须实现：

- 当前执行中的 Step 默认展开。
- 已完成 Step 默认折叠，只显示摘要。
- 用户点击 `▶ 开始 Step 1: 题目理解` 或 Step 标题，可以展开 / 收起。
- 每个折叠组显示事件数量、证据数量、状态。
- 错误、阻塞、等待用户确认的 Step 即使折叠，也要显示醒目状态。

推荐折叠标题：

```text
▶ Step 1：题目理解 · 已完成 · 6 events · 0 evidence
▶ Step 2：关键词拆解 · 已完成 · 5 events · 3 keywords
▼ Step 3：检索计划与候选证据 · 等待确认 · 8 events · 7 evidence
```

### 4.3 Trace 内容分层

展开后按类型显示：

- `系统状态`：开始、完成、暂停、用户确认。
- `证据事件`：检索到、标记可用、标记待核验、排除。
- `用户操作`：确认、修改、删除、恢复。
- `LLM 建议`：建议补充、建议排除、建议改写。

验收要求：

- 右侧 Trace 默认不会占满整屏。
- 用户能折叠 Step 1、Step 2 等历史记录。
- 当前 Step 的 Trace 自动展开。
- 折叠状态不影响 Trace 数据保存。

## 5. 左侧 LLM 对话入口

### 5.1 对话入口位置

左侧 `LLM 思维 / 对话` 区底部新增固定输入区：

```text
┌────────────────────────────┐
│ LLM 思维 / 对话             │
│ ...流式过程消息...          │
│                            │
├────────────────────────────┤
│ 和工作台讨论...             │
│ [输入框]                    │
│ [发送] [仅讨论] [生成修改建议] │
└────────────────────────────┘
```

输入区必须常驻，不随滚动消失。

### 5.2 对话能力范围

用户可以通过对话做这些事情：

- 询问某一步为什么这样判断。
- 要求解释某条证据是否可靠。
- 添加一条用户自己的论文、数据集、工程链接或文字证据。
- 删除或排除某条证据。
- 修改关键词。
- 新增关键点。
- 修改当前 Step 的摘要。
- 让 LLM 重新分析当前 Step。
- 让 LLM 比较两个候选方向。
- 让 LLM 生成下一步建议。

示例输入：

```text
把“道路裂缝”改成“钢材表面缺陷”，重新拆关键词。
这条 GitHub 工程看起来太老了，帮我标记为不推荐。
把这篇论文加入用户希望使用的论文。
为什么你认为这个题目能做？
我想加一个创新点：轻量化部署，帮我放到开题建议里。
```

### 5.3 写操作必须预览确认

对话入口不能直接静默修改工作台。

所有增删改操作必须走：

```text
用户输入
→ LLM 解析意图
→ 生成修改预览
→ 用户确认
→ 应用修改
→ 写入 Trace
```

修改预览必须显示：

- 操作类型。
- 修改对象。
- 修改前内容。
- 修改后内容。
- 影响哪些 Step。
- 是否会让后续 Step 变成 stale。

示例：

```text
修改预览
操作：修改关键词
对象：Step 2 / 对象关键词
修改前：道路裂缝
修改后：钢材表面缺陷
影响：Step 3-5 需要重新生成

[确认修改] [取消] [只作为备注保留]
```

### 5.4 对话意图类型

建议定义统一的 Workspace Command：

```text
WorkspaceCommand
├─ intent: ask | create | update | delete | reject | restore | regenerate | compare
├─ targetType: step | evidence | keyword | keyPoint | reportSection | traceEvent
├─ targetId
├─ patch
├─ reason
├─ affectedSteps[]
├─ requiresConfirmation: true
└─ sourceMessageId
```

首批支持意图：

| 意图 | 说明 | 是否需要确认 |
|---|---|---|
| `ask` | 只讨论、不修改 | 否 |
| `create evidence` | 添加证据卡 | 是 |
| `update keyword` | 修改关键词 | 是 |
| `reject evidence` | 标记证据不推荐 | 是 |
| `restore evidence` | 恢复证据 | 是 |
| `create keyPoint` | 新增关键点 | 是 |
| `update step summary` | 修改步骤摘要 | 是 |
| `regenerate step` | 重跑当前步骤 | 是 |
| `compare` | 比较两个方向或证据 | 否，除非落库 |

## 6. 数据与状态约束

### 6.1 左右两栏常驻

左侧 LLM 对话和右侧 Trace 均为常驻面板。

规则：

- 中间翻页不清空 LLM 对话。
- 中间翻页不清空 Trace。
- LLM 对话由 `llmTimeline` 和 `workspaceCommand` 驱动。
- Trace 由 `traceEvents` 和 `evidenceEvents` 驱动。

### 6.2 对话修改后的 stale 规则

如果用户通过对话修改了前序内容：

- 修改 Step 1：Step 2-5 标记为 `stale`。
- 修改 Step 2：Step 3-5 标记为 `stale`。
- 修改 Step 3：Step 4-5 标记为 `stale`。
- 修改 Step 4：Step 5 标记为 `stale`。
- 修改 Step 5：导出区标记为 `needs_regeneration`。

用户必须看到：

- 哪些步骤受影响。
- 是否需要重新生成。
- 旧结果是否保留为历史版本。

### 6.3 删除不等于物理删除

为了可追溯，证据删除默认是软删除：

- `rejected`：不推荐。
- `archived`：归档隐藏。
- `deleted_by_user`：用户删除但 Trace 保留。

禁止：

- 无确认永久删除证据。
- 删除 Trace 历史。
- 删除后不记录原因。

## 9. 人工验收脚本

### 9.1 布局检查

1. 打开页面。
2. 输入题目并完成 Step 1-5。
3. 查看工作台。

通过标准：

- 左侧是 LLM 思维 / 对话。
- 中间是当前 Step。
- 右侧是证据 Trace。
- 下方没有 Step 1-5 重复大卡。
- Step 6 导出仍保留。

### 9.2 Trace 折叠检查

1. 在右侧点击 `Step 1：题目理解`。
2. 展开事件。
3. 再次点击收起。

通过标准：

- 可展开。
- 可收起。
- 当前 Step 默认展开。
- 历史 Step 默认折叠。

### 9.3 对话入口检查

1. 在左侧输入：`为什么这个题目能做？`
2. 再输入：`把对象关键词改成钢材表面缺陷`。

通过标准：

- 第一条只讨论，不修改数据。
- 第二条生成修改预览。
- 用户未确认前不修改。
- 用户确认后修改生效并写入 Trace。
