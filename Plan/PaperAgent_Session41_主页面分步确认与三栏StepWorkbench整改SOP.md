# PaperAgent Session 41 SOP：主页面分步确认与三栏 Step Workbench 整改

## 0. 本次结论

当前人工验收未通过。核心问题不是“模型不会分析”，而是主页面仍然保留了一次性生成式交互：

- 点击“开始判断能不能做”后，题目理解、关键词、证据、可行性、推荐、开题等内容仍会一次性铺满页面。
- 页面没有把任务拆成可确认的步骤，也没有横向翻页。
- 已存在的 Step Deck / 流式卡片能力没有成为主入口，像是一个旁路 Beta 页。
- Trace 与进度没有成为当前步骤的可见反馈，用户难以判断系统“现在做到哪一步、为什么停下、下一步要我确认什么”。

因此 Session 41 的主线是：**先改页面交互骨架，把选题流程从“一键全出”改成“逐步确认 + 横向翻页 + 三栏工作台”。**

## 1. 本次目标

把“开始判断能不能做”改造成一个人能跟得上的开题/选题工作台：

1. 用户输入题目后，点击开始。
2. 系统只进入第 1 步：题目理解。
3. 第 1 步流式输出完成后暂停，等待用户确认或修改。
4. 用户确认后，才进入第 2 步：关键词拆解。
5. 后续每一步都遵循同样逻辑，不能一次性全部跑完。
6. 页面支持横向翻页，用户可以切换查看前后步骤。
7. 页面采用三栏结构：左侧证据 Trace，中间当前步骤卡，右侧 LLM 思维/对话状态。

## 2. 必须纠正的现有问题

### 2.1 主按钮仍绑定旧流程

需要检查并整改：

- `apps/web/index.html`
- `apps/web/app.js`
- `apps/web/step_deck.js`
- `apps/web/styles.css`

重点确认“开始判断能不能做”按钮是否仍然触发旧的全量分析流程。

整改要求：

- `#btn-analyze` 不再直接触发一次性渲染全部分析块。
- `#result-grid` 不再作为默认分析结果容器。
- 点击主按钮后，必须进入 Step Workbench 的第 1 步。
- 旧版一次性页面可以保留为“经典视图”或开发调试视图，但不能作为默认用户路径。

### 2.2 Step Deck 不能只是 Beta 页

已有的 Step Deck 思路需要变成主流程，而不是隐藏在旁路入口。

整改要求：

- 主页面默认进入 Step Workbench。
- 原 `page-step-deck` 可以复用，但需要接入真实主按钮。
- 若当前 Step Deck 仍是 mock stream，本次允许保留 mock 数据，但必须把状态机、分页、确认闸门做真实。
- 后续再接入真实 LLM / 检索 / 证据卡。

### 2.3 Trace 不能只是右下角日志

Trace 要成为用户理解系统行为的关键区域。

整改要求：

- 左侧 Trace 必须显示当前步骤、开始时间、完成状态、等待用户确认状态。
- 左侧 Trace 必须显示搜集到了哪些可用证据、哪些证据待核验、哪些证据被排除。
- Trace 必须能显示“为什么暂停”。
- Trace 必须能显示“用户确认后进入了哪一步”。
- Trace 与证据进展放在同一个可见区域，不能隐藏在用户不易发现的位置。
- 右侧必须以流式传输方式显示 LLM 的工作过程，也就是模型正在理解、拆解、检索、判断还是等待用户确认。
- 右侧必须报出自己查找了哪些页面、调用了哪些 Skill / 工具、为什么需要这些证据。
- 右侧的体验要接近当前 Codex 对话：边做边解释，解释要短，不要一次性倾倒大段日志。

## 3. 新页面信息架构

### 3.1 三栏布局

桌面端采用 25% / 50% / 25% 三栏布局。

```text
┌────────────────────┬────────────────────────────────┬────────────────────┐
│ 证据 Trace 25%      │ 当前步骤工作区 50%               │ LLM 思维/对话 25%   │
│                    │                                │                    │
│ - 可用证据          │ - 当前 Step 卡片                 │ - 流式回答          │
│ - 待核验证据        │ - 当前步骤主产物                 │ - 当前思考阶段      │
│ - 执行 Trace        │ - 功能分页面板                   │ - 页面/Skill 调用   │
└────────────────────┴────────────────────────────────┴────────────────────┘
```

如果宽度不足，移动端改为：

1. 顶部：步骤进度与当前状态。
2. 中部：当前步骤卡。
3. 底部：LLM 思维/对话。
4. 证据 Trace 收进可展开抽屉。

### 3.2 关键约束：左右两侧不随翻页直接切换

横向翻页只控制中间步骤工作区。

左右两侧是常驻状态面板，不能因为用户点上一页或下一页就直接替换成对应步骤的静态内容。

必须遵守：

- 中间 `activeStepIndex` 改变时，左侧证据 Trace 不应被清空。
- 中间 `activeStepIndex` 改变时，右侧 LLM 思维/对话不应被重置。
- 左侧只由证据事件、检索事件、Trace 事件驱动更新。
- 右侧只由 LLM 流式消息、模型阶段、用户确认问题驱动更新。
- 翻页是查看行为，不是执行行为，也不是重置行为。
- 用户回看 Step 1 时，右侧仍然可以显示当前系统已经暂停在 Step 2。
- 右侧 LLM 思维流可以持续追加，但不能因为用户翻页而重新生成或伪造历史。
- 中间页面的信息密度要受控，不能把所有证据、日志、解释堆在一个长页面里。

推荐数据结构：

```text
workspaceState
├─ activeStepIndex      # 中间正在查看哪一页
├─ currentRuntimeStep   # 系统真实执行到哪一步
├─ steps[]              # 每个步骤的结果、状态、锁定情况
├─ evidenceTrace[]      # 左侧常驻证据与执行 Trace
├─ llmTimeline[]        # 右侧常驻 LLM 流式输出与思维阶段
├─ toolUseTimeline[]    # 右侧常驻页面访问、Skill 调用、工具调用摘要
└─ gateState            # 当前等待用户确认的问题
```

### 3.3 左侧：证据 Trace 区

用途：让用户知道系统搜集到了什么证据、哪些能用、当前执行到哪里。

必须包含：

- 当前真实执行步骤。
- 已搜集到的论文证据。
- 已搜集到的数据集证据。
- 已搜集到的工程项目证据。
- 可用 / 待核验 / 不推荐的证据分组。
- Trace 事件流。
- 暂停原因。

证据 Trace 事件示例：

- `Step 1 开始：题目理解`
- `识别方法关键词：YOLO`
- `识别任务关键词：检测`
- `识别对象关键词：道路裂缝`
- `Step 2 等待用户确认关键词`
- `检索到论文候选 6 条，其中 3 条待核验`
- `检索到数据集候选 2 条，其中 1 条可用`

不能出现：

- 翻到 Step 1 时左侧 Trace 被重置成 Step 1 的静态摘要。
- 翻到 Step 3 时左侧直接伪造 Step 3 证据。
- 未发生的检索事件提前显示。

### 3.4 中间：当前步骤工作区

用途：承载用户当前正在审查的步骤页面。

必须包含：

- 当前步骤标题。
- 当前步骤目标。
- 当前步骤主要产物。
- 当前步骤证据摘要。
- 当前步骤风险或待确认点。
- 横向翻页控件。
- “确认并进入下一步”按钮。
- “修改本步”按钮。

不能出现：

- 一次性把所有步骤结果塞进中间工作区。
- 用户尚未确认时自动进入下一步。

### 3.5 右侧：LLM 思维与对话区

用途：让用户知道 LLM 正在想什么、说什么、卡在哪里，并允许用户追问或调整。

必须包含：

- 当前模型阶段，例如：`理解题目`、`拆关键词`、`准备检索`、`判断可行性`、`等待确认`。
- 当前步骤的流式回答，必须边生成边显示。
- 当前正在查找的页面或来源。
- 当前调用的 Skill / 工具名称。
- 调用 Skill / 工具的目的。
- 模型本轮输出的摘要。
- 当前确认问题。
- 用户追问/修改输入框。
- “采纳这个理解”或“重新分析本步”的操作。

不能出现：

- 翻页后右侧 LLM 对话被清空。
- 翻页后右侧伪装成另一个步骤正在运行。
- 用户未确认时右侧继续输出下一步内容。
- 只在最终结果里一次性展示“查了哪些页面、用了哪些 Skill”。
- 把所有原始日志不分层地塞进右侧，导致用户看不清当前行动。

右侧流式消息建议分为三类：

```text
assistant_thought   # 面向用户的简短过程说明
tool_use            # 页面访问、Skill 调用、检索调用
gate_question       # 当前需要用户确认的问题
```

示例：

```text
[assistant_thought] 我先把题目拆成方法、任务和对象三部分。
[tool_use] 使用 keyword_parser：提取 YOLO / 检测 / 道路裂缝。
[assistant_thought] 现在需要确认“道路裂缝”是不是你的真实研究对象。
[gate_question] 这个理解是否正确？确认后我再进入关键词扩展。
```

### 3.6 横向翻页规则

- 用户可以查看已经完成的前序步骤。
- 用户可以查看当前步骤。
- 未解锁步骤可以显示标题和锁定状态，但不能显示结果。
- 翻页只是查看，不等于执行下一步。
- 翻页只改变中间区域。
- 左侧证据 Trace 与右侧 LLM 思维区保持常驻，并由事件流驱动更新。

中间步骤页可以带自己的局部摘要，但左右两侧是全局工作台状态，不属于某一页的附属内容。

### 3.7 中间页面信息密度约束

中间 50% 区域要像“当前工作台页面”，不是报告全文预览。

硬性要求：

- 每个 Step 默认首屏不需要向下滚动。
- 证据数量少时，必须把内容控制在一屏内。
- 证据数量多起来后，不能继续纵向堆叠，要拆成子功能页、Tabs、横向卡组或筛选抽屉。
- 中间页面只展示当前用户需要判断的核心内容。
- 详细日志、工具调用、Skill 使用记录放在右侧 LLM 思维流。
- 证据全集、证据状态、排除原因放在左侧证据 Trace。

推荐拆分方式：

```text
Step 3：检索计划与候选证据
├─ 当前页：检索计划摘要
├─ 子页 A：论文候选
├─ 子页 B：数据集候选
├─ 子页 C：工程项目候选
└─ 子页 D：待确认问题
```

通过这种方式，用户不需要在一个长页面里找重点。

推荐 Trace 文案：

- `收到题目：基于 YOLO 的 XXX 检测`
- `开始 Step 1：题目理解`
- `Step 1 流式输出完成，等待用户确认`
- `用户确认 Step 1，进入 Step 2：关键词拆解`
- `Step 2 已暂停，等待用户调整关键词`

## 4. 分步流程定义

本次先固定 5 个主步骤，后续可以扩展。

### Step 1：题目理解

输入：

- 用户原始题目。

输出：

- 题目所属方向。
- 任务类型。
- 可能研究对象。
- 可能技术路线。
- 题目中的模糊词。
- 需要用户确认的问题。

暂停点：

- 必须等待用户确认题目理解是否正确。

通过条件：

- 页面只显示题目理解，不显示关键词结果。
- 用户点击确认后才进入关键词拆解。

### Step 2：关键词拆解

输入：

- 原始题目。
- 用户确认后的题目理解。

输出：

- 方法关键词，例如 `YOLO`。
- 任务关键词，例如 `检测`。
- 对象关键词，例如 `XXX`。
- 替代关键词，例如 `目标检测`、`缺陷检测`、`实时检测`。
- 检索关键词组合。

暂停点：

- 必须等待用户确认关键词是否正确。

通过条件：

- 用户能新增、删除、改写关键词。
- 未确认时不能自动进入检索。

### Step 3：检索计划与候选证据

输入：

- 用户确认后的关键词。

输出：

- 论文检索式。
- 数据集检索式。
- 工程项目检索式。
- 初步候选论文 / 数据集 / GitHub 项目卡片。

暂停点：

- 必须等待用户确认检索方向是否合理。

通过条件：

- 每类证据至少有独立区域。
- 候选证据要能标注“可用 / 待核验 / 不推荐”。

### Step 4：可行性判断

输入：

- 已确认的题目理解。
- 已确认的关键词。
- 候选论文、数据集、工程项目。

输出：

- 能不能做。
- 为什么能做。
- 为什么可能做不了。
- 数据集风险。
- 复现风险。
- 工程复杂度风险。
- 创新点空间。

暂停点：

- 必须等待用户确认是否继续走开题建议。

通过条件：

- 不能只给结论，必须给证据理由。
- 不允许在没有证据的情况下直接推荐开题。

### Step 5：开题建议

输入：

- 前 4 步已确认内容。

输出：

- 推荐题目版本。
- 研究背景草稿。
- 研究内容草稿。
- 技术路线草稿。
- 可行性说明。
- 风险与备选方案。
- 初步审稿式低门槛检查。

暂停点：

- 用户确认是否生成正式开题报告草稿。

通过条件：

- 能追溯到前面每一步的证据和用户确认。
- 不再出现“一键生成但不知道依据是什么”的体验。

## 5. 状态机要求

每个步骤必须有明确状态。

```text
locked -> running -> paused_for_review -> approved -> completed
                  -> needs_revision -> running
                  -> failed
```

规则：

- `locked`：未解锁，不可执行。
- `running`：正在流式输出。
- `paused_for_review`：输出完成，等待用户确认。
- `needs_revision`：用户要求修改当前步骤。
- `approved`：用户确认通过。
- `completed`：步骤结果已固化。
- `failed`：步骤失败，需要用户或系统重试。

强制约束：

- 一个时刻只能有一个步骤处于 `running`。
- 未获得 `approved` 的步骤不能触发下一步。
- 横向翻页不能改变状态机。
- 用户返回前序步骤修改后，后续步骤必须标记为 `stale` 或要求重新生成。
- 左侧证据 Trace 不受 `activeStepIndex` 直接控制。
- 右侧 LLM 思维/对话不受 `activeStepIndex` 直接控制。
- 左右两侧只能由事件流、LLM 流、用户确认和证据更新驱动。

## 6. 实施任务

### Task 1：主入口改造

目标：

- 点击“开始判断能不能做”后进入 Step Workbench。

执行要求：

- 查找 `#btn-analyze` 当前绑定逻辑。
- 停止默认调用旧的一次性分析渲染。
- 将按钮绑定到 `startStepWorkbench(topic)`。
- 第一次点击后只创建 Step 1。

验收：

- 点击后不出现全部结果块。
- 页面默认定位在 Step 1。
- 右侧 LLM 思维/对话区开始流式输出题目理解。
- 左侧证据 Trace 记录进入 Step 1。

### Task 2：三栏页面骨架

目标：

- 建立 25% / 50% / 25% 工作台布局。

执行要求：

- 新建或改造主容器，例如 `#step-workbench`。
- 左栏：证据 Trace。
- 中栏：横向步骤面板。
- 右栏：LLM 思维/对话。
- 固定各栏最小宽度，避免文本挤压和重叠。
- 左右两栏为常驻面板，不能绑定中间页码直接切换内容。

验收：

- 1366px 宽度下三栏清晰可见。
- 1920px 宽度下不显得松散。
- 390px 移动端不重叠。

### Task 3：横向翻页

目标：

- 用户可以横向查看步骤。

执行要求：

- 使用 step index 管理当前页。
- 提供上一页 / 下一页按钮。
- 支持键盘方向键作为增强项。
- 可使用 CSS `scroll-snap` 或 transform 实现。

验收：

- 已完成步骤可以回看。
- 当前步骤可以查看。
- 未解锁步骤显示锁定态。
- 翻页不会触发 LLM 继续生成。
- 翻页不会清空左侧证据 Trace。
- 翻页不会重置右侧 LLM 思维/对话。

### Task 4：逐步确认 Gate

目标：

- 每一步必须等待用户确认。

执行要求：

- Step 1 结束后进入 `paused_for_review`。
- 显示确认按钮。
- 用户确认后才进入 Step 2。
- 用户修改后重跑当前 Step，不进入下一步。

验收：

- Step 1 未确认时，Step 2 没有结果。
- Step 2 未确认时，Step 3 没有结果。
- Trace 能记录每次确认。
- 右侧 LLM 思维流能记录每次暂停前的判断过程。

### Task 5：左侧证据 Trace 与右侧 LLM 思维同步

目标：

- 用户能看懂系统当前做到哪一步、搜集到了哪些证据、LLM 正在处理什么。

执行要求：

- 每次 step 状态变化都写入左侧 Trace。
- 每次证据新增、判定、排除都写入左侧证据 Trace。
- 每次 LLM 流式输出、阶段切换、等待用户确认都写入右侧 LLM 时间线。
- 每次页面访问、Skill 调用、工具调用都写入右侧工具调用流。
- 右侧必须显示当前模型阶段。
- 暂停原因必须同时在左侧 Trace 和右侧确认问题中可见。
- 右侧内容必须逐条追加，不能等整步结束后一次性写入。

验收：

- 用户不用看控制台就知道系统暂停在哪里。
- 左侧至少记录：开始、证据新增、输出完成、等待确认、用户确认、进入下一步。
- 右侧至少显示：当前模型阶段、流式回答、页面访问、Skill / 工具调用、确认问题。

## 10. 风险与处理

### 风险 1：旧逻辑和新 Step Workbench 并存导致状态混乱

处理：

- 主入口只允许走一个状态机。
- 旧逻辑保留为调试函数时必须隐藏入口。

### 风险 2：横向翻页被误当成执行下一步

处理：

- 翻页只改变 `activeStepIndex`。
- 执行下一步只能通过 Gate 确认按钮。
- 左右两侧不监听 `activeStepIndex` 做内容重置。

### 风险 3：左侧 Trace / 右侧 LLM 状态与真实状态不同步

处理：

- 每个状态变化必须统一通过 `setStepStatus()`。
- 左侧 Trace 由状态变化和证据事件自动追加，不允许散落在 UI 事件里手写。
- 右侧 LLM 思维由 LLM 事件和 Gate 状态自动追加，不允许由翻页事件伪造。

### 风险 4：流式输出仍然跑完所有步骤

处理：

- 流式控制器必须按 Step 分段。
- 每段结束后停止。
- 只有收到用户确认事件后才创建下一段请求。

### 风险 5：右侧 LLM 思维流变成噪音

处理：

- 右侧只写面向用户的过程说明，不展示冗长原始日志。
- 页面访问和 Skill / 工具调用用一行摘要表示。
- 每条消息必须服务于“用户理解当前系统在做什么”。
- 原始调试信息只进入开发日志，不进入用户主界面。

### 风险 6：中间页面再次变成长报告

处理：

- 给每个 Step 设置信息预算。
- 超过 3 张证据卡必须拆子页。
- 超过 400 字的说明默认折叠。
- 当前待确认问题必须始终位于首屏。

## 12. 面试解释口径

这次整改可以作为 Agent 项目的重要面试亮点。

推荐解释：

> 早期版本是典型的一键生成式工具，用户点击后系统会一次性输出大量内容。人工验收发现这种方式不适合毕业选题场景，因为用户需要逐步校正题目理解、关键词、证据和可行性判断。于是我把主流程改成了 Human-in-the-loop 的 Step Workbench：中间是可横向回看的步骤页，左侧是常驻证据 Trace，右侧是常驻 LLM 思维/对话状态。翻页只负责查看，不会重置左右两侧。这样既降低了认知负担，也让后续报告生成可以追溯到每一步证据、用户确认和模型状态。

可强调的技术点：

- 一次性生成改为分步状态机。
- 用户确认 Gate 防止模型自动跑偏。
- 左侧证据 Trace 提高可解释性。
- 右侧 LLM 思维状态以流式方式提高过程透明度，并能展示查了哪些页面、用了哪些 Skill / 工具。
- 横向步骤页降低信息密度。
- 中间页面按功能拆分，避免把报告全文提前堆给用户。
- 旧系统被降级为调试路径，主路径变成人机协作流程。
