# TopicPilot-CN 轻量化开题报告助手 TODO

> 目标：在现有 OneTopic MVP 的基础上，再往前收缩一层，做成一个“只提供 skill + 一个网页”的轻量化版本。
>
> 用户不需要自己安装一整套项目，也不需要外接复杂 Skill 生态；外部由 Codex / Claude 这类模型来操控，前台只保留一个像“酒馆”一样的网页入口，用来完成开题选题、证据检索、初步判断和报告骨架生成。

---

## 1. 方向定义

- [ ] 明确轻量化版本的定位：不是完整平台，而是“开题报告助手前台 + 可执行 skill 包”
- [ ] 明确用户对象：只想快速做选题判断、开题初稿和低门槛审核的研究生
- [ ] 明确使用边界：优先覆盖中文工科 / 计算机 / AI 方向，不追求全学科通用
- [ ] 明确运行方式：用户只接触网页，不要求本地部署整套后端服务
- [ ] 明确交互方式：外部模型通过 skill 指令驱动页面上的分析流程

---

## 2. 产品形态

- [ ] 保留一个极简网页入口
- [ ] 网页首页只保留一个核心输入框：题目 / 方向 / 可选约束
- [ ] 网页保留结果区：题目拆解、证据、可行性、推荐题目、工作包、开题骨架
- [ ] 网页保留 trace 区：让用户看得懂分析进度，而不是只看到模型输出
- [ ] 网页保留最少的人工确认按钮：继续、收缩、换题、生成报告
- [ ] 网页避免重型账号体系和复杂配置项

---

## 3. Skill 设计

- [ ] 设计一个最小可用 skill 目录
- [ ] 拆成可组合 skill，而不是把所有逻辑堆进一个超大 skill
- [ ] 先做 4 个核心 skill
  - [ ] 题目拆解 skill
  - [ ] 文献检索 skill
  - [ ] 数据集 / baseline 检查 skill
  - [ ] 开题骨架 / 审核 skill
- [ ] 每个 skill 都输出结构化结果，方便网页直接渲染
- [ ] skill 输出要带证据字段，避免只给结论不给依据
- [ ] skill 之间只通过统一 schema 交接，不互相依赖隐式状态

---

## 4. 外部模型操控层

- [ ] 定义 Codex / Claude 调用 skill 的标准流程
- [ ] 定义“用户提问 → 模型选择 skill → skill 产出 → 网页展示”的链路
- [ ] 设计一个统一的指令协议，避免不同模型各说各话
- [ ] 预留“手工切换模型”的能力，方便不同模型做对照
- [ ] 让外部模型只负责调度，不负责保存核心状态
- [ ] 将状态存储留在网页侧或轻量后端侧

---

## 5. 网页最小闭环

- [ ] 输入一个选题
- [ ] 自动拆题
- [ ] 自动出检索词
- [ ] 自动查论文 / 数据集 / baseline
- [ ] 自动给出能不能做的判断
- [ ] 自动给出收缩路线或推荐题目
- [ ] 自动生成开题骨架
- [ ] 自动生成低门槛模拟审核意见

---

## 6. 数据与证据

- [ ] 统一证据结构
- [ ] 每条结论都要能回溯到证据来源
- [ ] 证据分为论文、数据集、代码、指标、经验规则五类
- [ ] 支持“未验证”状态，禁止把未检索到的内容当事实输出
- [ ] 支持“证据不足”结论，而不是强行给通过
- [ ] 让网页能展示证据卡片，而不是只展示长文本

---

## 7. 开题报告能力

- [ ] 先支持报告骨架，不急着支持长篇全文生成
- [ ] 输出结构至少包含：
  - [ ] 研究背景与意义
  - [ ] 国内外研究现状
  - [ ] 研究问题
  - [ ] 技术路线
  - [ ] 实验方案
  - [ ] 进度安排
  - [ ] 风险与备选方案
- [ ] 支持 2 到 3 个工作包规划
- [ ] 支持论文目录映射
- [ ] 支持“保毕业 / 稳中求新 / 冲更高水平”三种策略

---

## 8. 低门槛审核

- [ ] 保留一个轻量审核器，不做顶会式严格审稿
- [ ] 审核维度只看题目边界、数据、baseline、工作量、表达清晰度
- [ ] 审核输出只给四档：通过、有条件通过、需修改、不建议
- [ ] 审核结果要给修改清单，而不是只给一句结论

---

## 9. 实现拆分

- [ ] 第一阶段：把现有能力收拢成可复用 skill 输出
- [ ] 第二阶段：做单页网页壳子，接收并展示 skill 输出
- [ ] 第三阶段：定义外部模型控制协议，让 Codex / Claude 可以直接操控
- [ ] 第四阶段：把多个 skill 串成一个轻量工作流
- [ ] 第五阶段：补最小可用的 trace、缓存和状态持久化
- [ ] 第六阶段：补 Playwright smoke 测试

---

## 10. 最小验收标准

- [ ] 用户只输入一个题目，就能启动分析
- [ ] 系统能拆出关键词、对象、任务、方法
- [ ] 系统能给出论文 / 数据集 / baseline 证据
- [ ] 系统能输出可行性判断
- [ ] 系统能输出收缩路线或推荐题目
- [ ] 系统能输出开题骨架
- [ ] 系统能输出低门槛审核意见
- [ ] 网页能承接整个流程，不要求用户安装完整项目
- [ ] 外部模型能通过 skill 驱动整个过程

---

## 11. 暂缓事项

- [ ] 暂不做完整平台化后台
- [ ] 暂不做复杂权限和多租户
- [ ] 暂不做大而全的技能市场
- [ ] 暂不做重型工作流编排系统
- [ ] 暂不把报告生成做成无边界长文写作
- [ ] 暂不追求全学科通用

---

## 12. 推荐执行顺序

1. 先把 skill 结构定下来，保证每个能力块都能单独调用。
2. 再做一个极简网页，把输入、结果和 trace 串起来。
3. 然后定义 Codex / Claude 的操控协议，让外部模型真正能“接管”流程。
4. 最后再补报告导出、审核、测试和归档。

---

## 13. 一句话目标

把 TopicPilot-CN 从“一个完整系统”继续收缩成：

> 一个网页 + 一组 skill + 一个外部模型操控层，就能帮用户判断题目能不能做、怎么收缩、怎么开题。
