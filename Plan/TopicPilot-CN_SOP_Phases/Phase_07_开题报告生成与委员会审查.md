# Phase 07：开题报告生成与委员会审查

> 阶段目标：根据已定稿的工作包和实验矩阵生成开题报告骨架，并进行规则化 / 委员会式审查。  
> 方法依据：毕业论文合集强调“开题不是孤立文档，而是毕业论文骨架”“开题报告要展现工作量、创新点、实验方案和风险预案”。  
> 技术落点：`ProposalDraft`、`CommitteeReview`、`CommitteeReviewGraph`。  
> 阶段输入：`WorkPackage`、`ExperimentMatrix`、`EvidenceLedger`、`RiskScore`。  
> 阶段产物：`06_proposal_review.md`、开题报告骨架、审查意见、修改清单。

---

## 1. 阶段定位

Phase 07 是从“研究设计”进入“开题材料”的阶段。

它不追求一次生成最终版全文，而是生成一份可审查、可修改、可继续扩展的开题报告骨架：

- 研究背景与意义
- 国内外研究现状
- 研究问题与目标
- 研究内容与技术路线
- 拟解决关键问题
- 预期创新点
- 实验方案与评价指标
- 可行性分析
- 进度计划
- 风险预案

同时，本阶段要模拟开题委员会最常问的问题：

- 这个题目为什么值得做？
- 现有方法有什么不足？
- 你的创新点在哪里？
- 你的数据和 baseline 从哪里来？
- 实验怎么证明有效？
- 做不出来怎么办？

---

## 2. 开题报告生成规则

### 2.1 必须来自证据链

开题报告中的关键结论必须绑定来源：

- 研究现状来自论文证据。
- 数据与实验来自数据集和指标证据。
- baseline 来自代码 / 方法证据。
- 创新点来自工作包和实验矩阵。
- 风险预案来自 Phase 05 风险评分。

### 2.2 创新点表达

创新点不能写成空泛口号。

推荐模板：

```text
针对现有开题选题过程中 {问题}，本文设计 {方法/模块}，
通过 {实验/评价方式} 验证其在 {指标/场景} 上的有效性。
```

### 2.3 工作量表达

开题报告中要突出：

- 文献调研工作量
- baseline 与实验可行性评估工作量
- 风险评分与 Pivot 决策工作量
- 工作包与实验矩阵设计工作量
- 开题报告结构化生成工作量

---

## 3. 委员会审查维度

| 审查维度 | 审查问题 | 不通过表现 |
|---|---|---|
| 题目边界 | 是否过大、过泛、不可验证 | 通用、全自动、全场景 |
| 研究现状 | 是否有文献分类和不足总结 | 只堆论文，不分类 |
| 创新点 | 是否绑定问题、方法、实验 | 只有概念，没有验证 |
| 数据与 baseline | 是否可获得、可复现、可比较 | 没数据、没代码、没指标 |
| 实验方案 | 是否有主实验和消融 / 案例 | 只有主观描述 |
| 工作量 | 是否能支撑第三章、第四章 | 只有一个功能 demo |
| 风险预案 | 失败时是否能收缩或替代 | 无 fallback |

## 5. 当前验收标准

- [ ] 开题报告骨架包含 10 个必要部分。
- [ ] 国内外研究现状按类别组织，而不是简单罗列论文。
- [ ] 每个创新点绑定问题、方法和实验。
- [ ] 实验方案包含数据、baseline、指标和主实验。
- [ ] 风险预案能回应 Phase 05 的最高风险项。
- [ ] 委员会审查至少覆盖 7 个维度。
- [ ] 输出追问清单和修改清单。
- [ ] 不允许出现没有证据来源的强结论。

## 6. 界面 MVP 后置验收

界面出现后，应补充：

- 用户能勾选修改项。
- 每个创新点能展开查看证据来源。
- 页面能导出 Markdown 初稿。
