# Phase 01：任务建档与毕业目标确认

> 阶段目标：把“学生想做什么”转换为可持久化、可追踪、可评估的项目建档信息。  
> 方法依据：毕业论文合集强调“选题首先是目标管理”，先确定毕业目标与风险上限，再谈题目优劣。  
> 技术落点：FastAPI + Pydantic v2 建立 `ProjectIntake` / `TopicIntakeState`，作为 LangGraph 初始状态。  
> 阶段输入：学生原始想法、导师要求、学院要求、时间节点、资源条件。  
> 阶段产物：`00_input.md`、`ProjectIntake` 结构化对象、项目记录。

---

## 1. 阶段定位

Phase 01 不做题目推荐，也不急着检索论文。它只回答一个问题：

> 这个毕业任务的约束是什么，后续 Agent 应该在什么风险边界内行动？

毕业论文合集中的核心经验是：保毕业、稳中求新、冲高水平是三套不同路线。若一开始不确认目标档位，后续容易出现“题目听起来高级，但没有数据、没有 baseline、没有第一张实验表”的问题。

TopicPilot-CN 在本阶段要完成三件事：

- 形成学生、导师、学院、时间、资源五类约束画像。
- 明确题目可接受风险，尤其是第一张实验结果表的时间红线。
- 把所有缺失信息显式记录为待补项，禁止隐式假设。

---

## 2. 毕业方法论转化

| 毕业论文合集经验 | Phase 01 中的产品化规则 |
|---|---|
| 方向选择是目标管理 | 先问目标档位，再进入题目拆解 |
| 保毕业选“人多方向” | 目标为保毕业时，后续优先成熟方向、公开数据、可复现代码 |
| 继承资源很重要 | 必须记录同门论文、课题组代码、已有数据和导师项目 |
| 第一张结果表是生死线 | 必须记录最晚出结果表时间，并作为风险评分约束 |
| 不要造航母 | 若目标低风险但资源不足，后续必须触发收缩或 Pivot |

## 3. 技术设计

### 3.1 Pydantic 对象

```python
class ProjectIntake(BaseModel):
    case_id: str

    major: str | None
    degree_type: Literal["本科", "硕士", "博士", "未知"]
    goal_level: Literal["保毕业", "稳中求新", "冲高水平"]
    thesis_deadline: str | None
    proposal_deadline: str | None
    first_result_deadline: str | None
    advisor_direction: str | None
    school_requirements: list[str]

    student_resources: StudentResourceProfile
    raw_topic: str
    must_keep: list[str]
    can_drop: list[str]
    missing_fields: list[MissingField]
    intake_rating: Literal["A", "B", "C", "D"]
```

### 3.2 API 落点

```text
POST /api/v1/projects
GET  /api/v1/projects/{project_id}
POST /api/v1/projects/{project_id}/intake/validate
```

### 3.3 LangGraph 状态

Phase 01 是整个 `TopicPilotGraph` 的入口节点：

```text
IntakeNode
→ IntakeValidationNode
→ HumanClarificationNode? 
→ TopicDecompositionNode
```

如果 `intake_rating` 为 C 或 D，流程进入人工补问，不允许进入 Phase 02。

## 5. 执行步骤

### Step 1：建立案例编号

格式：

```text
YYYYMMDD_专业简称_题目关键词
```

示例：

```text
20260616_AI_开题选题助手
```

### Step 2：确定目标档位

目标档位决定后续策略：

| 目标档位 | 后续策略 | 禁止路线 |
|---|---|---|
| 保毕业 | 成熟方向、公开数据、可复现 baseline、章节模板充分 | 原始创新、从零采数、从零训练大模型 |
| 稳中求新 | 成熟任务 + 局部改进 + 可解释实验 | 多个高风险创新点同时推进 |
| 冲高水平 | 可探索新问题，但必须保留保底工作包 | 没有 fallback 的单一路线 |

### Step 3：确认可继承资源

优先追问：

- 有没有同门论文目录可以参考？
- 有没有师兄师姐代码或数据可以合法复用？
- 有没有导师项目中的真实场景或数据？
- 有没有已运行过的 baseline 环境？

如果没有继承资源，Phase 03 和 Phase 04 必须把“找公开 baseline”和“找公开数据”作为最高优先级。

### Step 4：确认时间红线

必须记录：

- 距离开题还有多久。
- 距离毕业还有多久。
- 最晚什么时候必须有第一张主结果表。

若时间红线小于 baseline 复现预估周期，则后续必须触发题目收缩。

### Step 5：输入完整性评级

```text
A：输入完整，可直接进入 Phase 02。
B：基本可用，可进入 Phase 02，但必须带显式假设。
C：关键缺失，必须补问后再进入 Phase 02。
D：当前不可评估，题目、目标、时间或资源严重缺失。
```

## 7. 验收标准

### 7.1 文件验收

- [ ] 已创建 `00_input.md`
- [ ] 包含 9 个必要部分
- [ ] 目标档位明确
- [ ] 开题时间、毕业时间、第一张结果表时间至少有两个明确
- [ ] 导师方向、学院要求、资源条件有记录
- [ ] 缺失项进入“待补问题”，没有隐式假设

### 7.2 数据结构验收

- [ ] `ProjectIntake` 可通过 Pydantic 校验
- [ ] `goal_level` 不为空
- [ ] `raw_topic` 不为空
- [ ] `missing_fields` 可被后续补问节点读取
- [ ] `intake_rating` 为 A/B 时才允许进入 Phase 02

### 7.3 阻断条件

出现以下情况必须阻断：

1. 只有一句“帮我想个题目”，没有专业、导师方向或毕业时间。
2. 目标档位是保毕业，但坚持从零采集、标注、训练和部署。
3. 用户要求编造论文、数据或实验结果。
4. 资源条件全部未知，却要求判断能否毕业。
5. 第一张结果表时间已经无法支撑当前题目规模。
